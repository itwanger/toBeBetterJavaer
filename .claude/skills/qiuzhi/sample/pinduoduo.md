## 派聪明面试题回答集锦

### 1、先 KNN 再关键词取交集，是否会漏掉文档？

会的。用户搜"如何提升代码质量"，向量检索召回了一篇讲"重构最佳实践"的文档，语义高度相关。但 BM25 那边因为没有"代码质量"这几个字，召回不了。取交集就把这篇相关文档丢了。

![](https://cdn.tobebetterjavaer.com/paicoding/89be37ffbfdc82a4b1ce8b7fb98315dc.png)

用户搜索一个特定的错误码 AX-123 ，这个错误码可能出现在一篇非常长的、主题是“系统维护”的文档中。这篇文档的整体语义和“错误码”可能相去甚远，导致它在 KNN 召回中排名很低，甚至根本没被召回。那么，取交集自然也就找不到它了。

### 2、BM25 和 KNN 得分量纲不一致，加权计算是否不合适？

BM25 的得分是一个相对值，它没有固定上限，可以从零到几十甚至上百。得分高度依赖于词频（TF）、逆文档频率（IDF）等统计特征。

![](https://files.mdnice.com/user/3903/9fe25fb4-cc12-4759-ae19-afb815c29e96.jpg)

KNN（余弦相似度）的得分通常在 `[0, 1] 或 [-1, 1]` 的区间内。

简单的加权求和在量纲不一致的情况下确实有待改进。

### 3、通过 RRF 算法计算得分并排序是否好一些？

RRF (Reciprocal Rank Fusion, 倒数排名融合) 的核心思想是：“一个文档越是在多个不同的排名列表里都排在靠前的位置，它的最终得分就越高。”

计算公式为：`RRF_Score(doc) = Σ (1 / (k + rank_i))` rank_i 是文档 doc 在第 i 个排名列表中的排名（比如在 BM25 结果中排第 3，在 KNN 结果中排第 10）。

![](https://files.mdnice.com/user/3903/e98aec79-2b87-46f4-bfe8-9e4d185ceb6a.png)

k 是一个常数（通常设为 60），用于防止排名靠后的文档对总分影响过大。 RRF 能够解决量纲不一致的问题。无论 BM25 分数是 100 还是 1000，只要它排第一，贡献的都是 1 / (k + 1) 。

面试中，如果能自信地说出 BM25 和 KNN 分数量纲不一致的问题，并提出使用 RRF 作为解决方案，会是一个巨大的加分项。

### 4、面试时说“手动计算了 RRF 得分”，是否存在漏洞？

向 Elasticsearch 发送一个只包含 BM25 的查询，拿到一个排名列表。再发送一个只包含 KNN 的查询，拿到第二个排名列表。然后再写一个循环，遍历两个列表，根据 RRF 公式手动计算每个文档的最终得分，重新排序。

![](https://cdn.tobebetterjavaer.com/paicoding/b75994235c9d8dc82bb852e33648d782.png)

Elasticsearch 8.x 之后的版本，已经原生支持在一次查询中同时定义 knn 和 query （BM25），并直接指定使用 rrf 作为融合方法。这意味着可以只发送一次请求 ，ES 会在内部帮你完成所有事情，然后直接返回最终的、融合排序后的结果。

综合回答。 “在我项目的早期版本中，由于考虑的不充分，我确实分别执行了 KNN 和 BM25 查询，在下个版本中，我的优化方向是将 RRF 的计算下推到搜索引擎层，在一次请求中完成混合搜索和结果融合。”
