二哥，派聪明项目中，LLM回答内容中包含RAG的文档引用信息，我看是根据固定格式 (来源#数字: 文件名)进行匹配解析，再根据fileName进行的文档下载（区分了有无userToken的情况），但是如果fileName存在重复，那就找到多个文档，因为用的是findFirst，那就有可能找到的同名文档并不是原来从RAG中检索到的。


首先，这块确实是存在漏洞的，没有做的尽善尽美，派聪明中其实还有不少这样的问题，一直想要解决，但都因为我个人时间有限，在耽搁着。

- 用户A 在 3 月 1 日上传了一个文件，名为项目周报.pdf ，其内容是关于“市场部”的。系统为其生成了一个唯一的 ID（比如 MD5 哈希值 AAA ）。
- 用户A 在 3 月 8 日又上传了一个文件，也叫项目周报.pdf ，但其内容是关于“研发部”的。系统为其生成了另一个唯一的 ID（ BBB ）。
- 现在，用户提问：“上周研发部有什么进展？”
- RAG 系统非常准确地从文件 BBB （研发部周报）中找到了相关内容，并生成了答案。
- LLM 在答案末尾按要求添加了引用： “...以上内容参考自 (来源#1: 项目周报.pdf)” 。
- 用户点击这个引用链接，想要下载原文。
- 后端收到请求，解析出 fileName = "项目周报.pdf" 。
- 后端去数据库或文件索引中执行查询： findByFileName("项目周报.pdf") 。
- 数据库中存在两个同名文件（ AAA 和 BBB ）。由于数据库的内部排序和 findFirst() 的不确定性，这次查询碰巧返回了文件 AAA （市场部周报）。
- BUG 出现 ：用户下载到了一个错误的源文件，因为下载的文档内容和 AI 的回答完全对不上。

解决这个漏洞的思路是：在整个“检索-生成-引用”的链条中，始终传递文件的唯一标识符（Unique ID） ，而不是可能重复的 fileName 。这个唯一 ID 可以是文件的 MD5 哈希，也可以是数据库自增的 fileId 。

第 1 步：改造 RAG 检索与 Prompt 构建

- 当 HybridSearchService 检索到相关的文档分片时，它不仅要获取分片的文本内容，还必须同时获取该分片所属文件的唯一 ID ( fileId 或 fileMd5 ) 和 文件名 ( fileName ) 。
- 在构建最终喂给 LLM 的 Prompt 时，上下文的格式需要改变，新的 Prompt 上下文格式（包含唯一 ID） ：

...
[来源1, ID: 'BBB', 文件名: '项目周报.pdf']
研发部上周的主要工作是完成了支付模块的重构...
...

第 2 步：修改对 LLM 的指令

在 System Prompt 中，对 LLM 生成引用的指令也需要更新。

当你引用来源时，你必须严格使用以下格式：(来源#数字: 文件名, ID: 唯一ID)。例如：(来源#1: 项目周报.pdf, ID: 'BBB')。

第 3 步：改造后端的引用解析和下载逻辑

- 当 LLM 返回包含新格式引用的答案时，后端的解析逻辑也要随之升级。它不再是简单地用正则表达式匹配 (来源#数字: 文件名) ，而是要匹配 (来源#数字: 文件名, ID: 唯一ID) 。
- 解析后，它会同时得到 fileName （用于在前端显示）和 fileId （用于后端查找）。
- 当用户点击下载链接时，前端将这个唯一的 fileId 发送给后端。
- 后端的下载接口，从 findByFileName(fileName) 改为 findById(fileId) 。
