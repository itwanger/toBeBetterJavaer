# AI技术文章生成工作流

## 环境声明（每次执行前必读）

执行本工作流前，先运行以下命令获取当前真实日期：

```bash
date "+%Y年%m月%d日"
```

后续所有涉及日期的操作必须基于这个日期：联网搜索关键词带上当前年月，文章 `date` 字段使用当前实际日期，正文时间描述基于当前日期。

---

## 概述

根据指定选题，按照二哥的写作风格（references中的任意一篇）完成AI技术类文章撰写。定位是AI博主/AI求职博主。

## 工作流程

### 步骤0：选择参考文章（强制第一步）

**每次执行任务前，必须先让用户选择参考文章。**

使用 `AskUserQuestion` 工具询问用户参考哪一篇历史文章，提供以下选项：

| 文件名 | 文章标题 | 风格特点 | 适用场景 |
|--------|----------|----------|----------|
| `glm4-7.md` | 实测GLM-4.7，我想给他当股东了 | 产品测评类，技术深度高，强调有意思、越来越有意思，ending 情绪升华强调 AI 编程的未来 | AI工具实测、大模型测评、能力对比分析 |
| `quest-2.md` | 转发很高 | 工具介绍+实战教程，口语化强（啧啧啧、爽歪歪），强调学习，从现在开始，有简历包装环节 | AI工具教程、实战项目、技术栈集成 |
| `OpenClaw-install.md` | OpenClaw 安装教程，全网最详细手把手教你接入飞书！ | 安装教程类，保姆级手把手教学，大量截图和问题排查，手摸手那种、方便得很 | 安装配置教程、接入指南、常见问题排查 |
| `OpenClaw-unstall.md` | OpenClaw 卸载，引入面试题！ | 内容扎实、热点、面试官喜欢问 | 转发率高、点赞高 |

用户选择后，**必须先读取对应的 reference 文件，学习其写作风格**，然后才开始后续步骤。

### 步骤1：检查素材

`./sucai.md`（默认主素材）

你必须精读素材库中的内容，消化吸收。读取后提取关键信息、数据、观点、截图，作为正文素材池，尤其是截图。我们要写的内容是基于素材完成的，不脱离，但有创新。

**不要只做“读过了”的口头确认，必须先把素材拆出来。**

至少提取这 5 类素材：
- 选题钩子：适合放在标题、开头、前言冲突里的句子或事实
- 明确观点：`sucai.md` 已经给出的判断、方向、章节范围
- 事实与数据：阅读量、时间、版本号、提交数、文件数、发布时间
- 链接与出处：官方文档、上一篇文章、社区讨论、截图地址
- 截图与证据：`sucai.md` 里已有的图片、日志、页面、命令输出

如果 `sucai.md` 很短，也不能跳过这一步。越短，越要把里面每一条高信号信息吃干榨净。

### 步骤1.5：整理素材映射表（强制执行）

正式搜索和写作前，必须先输出一张“素材映射表”，把 `sucai.md` 里的高信号素材逐条安排到正文里。

高信号素材默认包括：
- 明确给出的截图或图片链接
- 明确给出的上一篇文章链接或历史文章链接
- 明确给出的官方文档链接
- 明确给出的数据、时间、版本号、阅读量
- 明确点名要展开的主题词

推荐格式：

```markdown
素材映射表

| 素材项 | 类型 | 计划放入章节 | 使用方式 | 如果不用，理由 |
|--------|------|--------------|----------|----------------|
| 7000多阅读 | 数据 | 前言 | 用来承接上一篇反响，制造选题合理性 | |
| sucai里的截图链接 | 截图 | 前言/## 01 | 直接复用或改成截图占位符 | |
| 上一篇文章链接 | 外链 | 前言 | 作为上下文承接 | |
| 飞书多应用接入 | 主题 | ## 01 | 作为第一章主标题 | |
| 音频 | 主题 | - | 本文篇幅不够，降级到下篇选题 | 与本篇主线不够强相关 |
```

规则：
- `sucai.md` 里的每一项高信号素材，都必须在映射表里出现。
- 允许不用，但不能“悄悄不用”。不用就写明原因。
- 没有完成素材映射表，不允许进入步骤2和步骤4。

### 步骤2：搜集资料

用联网搜索（如 web_search）搜索该选题的相关资料，关键词带上当前日期，确保信息时效性。

搜索关键字参考“AI 最新进展”、“大模型 本周 发布”、“AI应用 最新动态”、“国产大模型 近期 新闻”等关键词，必须带上时间限定。默认只采用最近7天内的信息，超过7天的内容仅可作为背景，不得作为核心结论证据。

补充要求：

- 可补充引用的公开跑分/基准信息（如公开榜单、官方基准、第三方测试）。
- 补充来自 X（原 Twitter）的外界评价，优先真实开发者或有实测记录的账号。
- 外部引用必须保留来源链接和日期，避免“听说”“网友表示”这类模糊归因。

### 步骤3：整理证据清单（先于写作）

写正文前，先整理“引用证据清单”，至少包含：`结论点`、`来源链接`、`发布时间/发帖时间`、`为何可信`。

如果未检索到可核验的跑分或 X 外界评价，必须在清单里明确标记“未检索到有效证据”，正文对应段落降级为“经验观察”，禁止伪造数据或伪造引用。

如果一个结论点本身来自 `sucai.md` 提供的官方文档、上一篇文章、截图或数据，必须在证据清单中标注“来源于 sucai.md”，不能把素材来源抹平。

### 步骤3.5：章节大纲与字数预算（强制执行）

正式写正文前，必须先列“章节大纲 + 字数预算”。不要直接开写。

最低要求：
- 前言 1 段，预算 300 到 500 字
- 核心章节至少 4 个，按 `## 01` 到 `## 04` 编号
- 每个核心章节预算至少 800 字
- `## ending` 预算至少 300 字

**预算总和不能只卡 4000，必须做到至少 4200 字。** 因为模型落稿时通常会比预算再缩一点，4200 是安全线，不是上限。

推荐格式：

```markdown
写作蓝图

- 前言：为什么要测这个选题；预计 400 字
- ## 01、xxx：第一轮实测过程和核心发现；预计 900 字
- ## 02、xxx：和竞品/旧方案对比；预计 900 字
- ## 03、xxx：踩坑、失败样例和修复过程；预计 900 字
- ## 04、xxx：适合谁用，不适合谁用，怎么落地；预计 900 字
- ## ending：情绪升华和价值判断；预计 350 字
```

如果预算不足 4200，先补章节或补内容维度，再开始正文。不要把“补字数”这件事留到最后。

### 步骤4：撰写文章

文件格式为 Markdown，**正文最少 4000 字，推荐控制在 4500 到 6500 字之间**。

不要写“4000字左右”这种模糊目标，这会让模型默认收缩。写作时先给每一部分分配篇幅，避免最后只写成 2000 多字。

推荐配额：
- 前言：300 到 500 字
- 每个核心章节：700 到 1200 字
- ending：不少于 200 字，推荐 300 到 500 字

如果正文结构低于 4 个核心章节，通常很难稳定写到 4000 字。默认补足章节，或者补充案例、对比、截图解读、踩坑过程、失败样例。

写作时必须按步骤3.5的字数预算推进。如果某一章明显短于预算，不要急着收尾，优先补这几类内容：
- 这一步到底怎么做的
- 中间失败了什么
- 和其他方案差在哪
- 截图里到底能看出什么
- 读者照着做时最容易踩什么坑

正文必须显式消化步骤1.5里的素材映射表，不能只借 `sucai.md` 的题目，不借它的内容。

最低使用要求：
- `sucai.md` 里的截图链接、历史文章链接、官方文档链接，至少命中其中 2 类进入正文或证据清单。
- `sucai.md` 里列出的主题词，正文至少覆盖 70%。如果故意不写某一项，要在素材映射表里写原因。
- 前言必须至少吸收 1 个来自 `sucai.md` 的具体素材，不能只写通用开场白。

**注意：标题字段先填写临时标题（如临时标题），待步骤5确认后再更新。**

文章头部模板：
```yaml
---
title: 临时标题
shortTitle: 短标题
description: 文章描述
tag:
  - Agent
category:
  - AI
author: 沉默王二
date: # 使用 date 命令获取的实际日期，格式 YYYY-MM-DD
---
```

### 步骤5：生成标题选项（强制执行）

文章正文完成后，根据 `./biaoti.md` 的风格，任选 5 种不同风格生成标题选项，使用 `AskUserQuestion` 工具供用户选择。

**重要：严格按照 biaoti.md 的用词和句式模式，不要随意发挥。**

参考 biaoti.md 的 6 种具体模式（包含实际用词示例）：

| 模式 | 实际示例 | 关键用词/句式 |
|------|----------|--------------|
| **模式1：数据+热词** | 这份Claude Code指南火爆全网，已狂飙20k+ Star！ | 火爆全网、狂飙、爆火、又一个...爆火、37K+星标！ |
| **模式2：悬念+行动** | DeepSeek V4要来？我花了1个小时，用神秘模型Pony开发了一个macOS应用，丝滑切换Claude Code底层模型 | 要来？、我花了...、丝滑切换 |
| **模式3：拉踩+体验** | 不用Claude，这个国产Cowork就很猛，我测了一天，直呼太香！ | 不用...、就很猛、我测了一天、直呼太香！ |
| **模式4：范围+发现** | Top 10热门Agent Skills，我试了个遍，发现真的能让生产力翻倍、马斯克开源 X 推荐算法，我研究了一天，发现了这些宝藏设计 | Top 10、试了个遍、发现真的能、研究了一天、发现了这些... |
| **模式5：夸张+功能** | 阿里杀疯了，桌面端 Agent 强势来袭，自动做 PPT，视频，太牛了。 | 杀疯了、强势来袭、太牛了、有点东西 |
| **模式6：对话怼人** | “桃厂面试官：‘Agent 不就是 LLM 挂几个插件？’，我反问：‘Agent Loop、上下文工程、MCP，您管这叫挂插件？’” | 面试官：'...'、我反问、我怼回去、您管这叫...、你确定？ |

**禁止词汇**：这些 biaoti.md 中没有出现过的词不要用
- ❌ 震撼、惊艳、炸裂（ biaoti.md 没有）
- ❌ 必看、必读、干货（ biaoti.md 没有）
- ❌ 全网首发、史上最强（ biaoti.md 没有）
- ❌ 神级可以（第8条有又一个神级）

**推荐用词（从 biaoti.md 提取）**：
- 体验类：有点东西、太香了、太牛了、直呼太香、真猛
- 行动类：我测了一天、试了个遍、研究了一天、一口气给你
- 发现类：发现真的能、发现了这些、一文带你看懂

用户选择最终标题后，更新文章的 `title` 字段。

### 步骤6：落盘输出

文件命名用文章标题关键词，保存到 `docs/src/sidebar/itwanger/ai/` 目录（相对仓库根目录）。

### 步骤6.5：正文长度校验（强制执行）

正文写完并落盘后，**先不要交付**。必须执行下面的脚本统计正文长度：

```bash
python3 .claude/skills/ai-article/scripts/check_body_length.py <文章文件路径> --min 4000
```

规则：
- 统计对象只包含正文，排除 YAML frontmatter、代码块、行内代码、URL、Markdown 链接地址。
- 中文按字统计，连续英文单词按 1 个词计数。
- 结果小于 4000，视为未完成，必须继续扩写后重新执行。

**扩写优先顺序**：
- 补充实测过程，而不是灌水感受。
- 补充对比细节，而不是重复结论。
- 补充截图解读、踩坑过程、失败样例、边界条件。
- 补充读者可直接复用的方法、提示词、命令、配置思路。

### 步骤6.6：正文标点归一化（强制执行）

正文写完并落盘后，**不要相信模型已经完全遵守中文标点规则**。必须执行下面的脚本，对正文做一次机械归一化：

```bash
python3 .claude/skills/ai-article/scripts/normalize_zh_punctuation.py <文章文件路径>
```

这个步骤是格式兜底，不是可选优化。尤其是 Claude 这类模型，常常会在中文正文里混入英文双引号、英文冒号，或者在中文句子里给英文产品名两边加空格。脚本会只处理正文，跳过 YAML frontmatter、代码块、行内代码、URL 和 Markdown 链接。

### 步骤7：交付前检查（强制执行）

文章完成后，必须逐项检查以下清单，未完成的必须修改后再交付：

- [ ] 正文使用中文标点符号（逗号必须是中文逗号 `，`，双引号必须是中文双引号 `“xxx”`）
- [ ] 已完成“素材映射表”，且 `sucai.md` 的高信号素材都已安排去向
- [ ] 写正文前已先完成“章节大纲 + 字数预算”，预算总和不少于 4200
- [ ] 落盘后已执行 `check_body_length.py`，正文长度不少于 4000 字
- [ ] 落盘后已执行 `normalize_zh_punctuation.py`，而不是只靠模型自觉输出中文标点
- [ ] 标点检查仅针对正文，排除代码块、行内代码、URL、YAML frontmatter、命令行
- [ ] 二级标题格式为“## 01、标题”、“## 02、标题”
- [ ] 每个核心章节（## 01及之后）至少1个截图占位符
- [ ] 每个截图占位符包含“截图目标”和“关键词”
- [ ] 结尾用## ending，有情绪升华（不低于200字）
- [ ] 外部结论都有来源链接和日期，且在“引用证据清单”中可回溯
- [ ] `sucai.md` 中的截图、链接、数据、主题词，没有被无声丢弃；不用的部分已在素材映射表中说明原因
- [ ] 避免 AI 味词汇（值得注意的是、此外、标志着等）
- [ ] 避免互联网黑话（赋能、抓手、闭环、打通等）
- [ ] 正文最少 4000 字，推荐 4500 到 6500 字

## 写作原则

### 标题风格（最高优先级，强制执行）

套用 `./biaoti.md` 中的任一标题风格，如果你不确定，可以生成5个供我选择。

参考标题特点：
- 直接、不绕弯，一句话讲清核心卖点
- 避免学术化、正式化表达

### 语气和称呼

开头固定用“大家好，我是二哥呀。”，用大家、我们、小伙伴和读者拉进关系，保持对话感。语气像老朋友聊技术，不是教科书，要有温度、有态度。

### 标点符号（最高优先级，强制执行）

**正文必须使用中文标点符号，这是硬性要求。**

**这条规则不能只靠提示词约束，必须配合步骤6.6的脚本做落盘后的机械校正。**

具体要求：
- 中文逗号：， （英文逗号 , 禁止使用）
- 中文句号：。
- 中文冒号：
- 中文分号；
- 中文引号：“” （英文引号 " 禁止使用）
- 中文括号：（）
- 中文问号：？
- 中文感叹号：！

**例外场景**：代码块、行内代码、URL、YAML frontmatter、命令行、外文原文引用中的标点保持原样。

**生成后检查步骤**：文章生成完成后，检查正文中的逗号是否为中文逗号（全角 `，`）、双引号是否为中文双引号（全角 `“”`）；例外场景不做替换。

**中文正文里的中英混排规则**：
- 中文句子里，英文工具名、模型名、库名两侧默认不加空格。写 `卸载过OpenClaw吗？`，不要写 `卸载过 OpenClaw 吗？`
- 对话句优先写成“人物动作 + 中文冒号 + 中文双引号”的形式，避免生硬的台词标签。

示例：
- 错误：面试官："卸载过 OpenClaw 吗？"
- 正确：老王开门见山：“卸载过OpenClaw吗？”

### 历史文章参考风格（重要）

**在步骤0用户选择参考文章后，必须精读该文件，学习二哥的写作风格。**

不要只看 SKILL.md 的规则，要看实际例子。重点关注：

- **开头套路**：如何用第一段就抓住读者注意力
- **标题风格**：参考对应的 `./biaoti.md`，但也要看实际文章标题是怎么命名的
- **章节结构**：二级标题"## 01、xxx"的节奏和内容分配
- **截图密度**：每个章节配图的时机和方式
- **口语化表达**："说真的"、"讲真"、"啧啧啧"、"爽歪歪"等词的使用场景
- **ending 升华**：如何从技术内容自然过渡到情绪价值

- 避免纯标题党，标题承诺的内容，正文必须能兑现。
- 避免过度营销，有新意，有人文情怀，能引发共鸣的标题更有吸引力。

### 文章开头套路

开头要能抓住读者注意力，但不要标题党。

### 前言冲击力（强制）

前言必须在前3段内完成“冲突-结果-收益”三连：

- 冲突：明确一个真实痛点或争议点。
- 结果：提前给出一句高价值结论（但不剧透全部细节）。
- 收益：告诉读者继续读下去能拿到什么（方法、结论、避坑点）。

推荐句式：

- 我原本以为 X，结果实测后发现 Y。
- 真正拉开差距的不是 A，而是 B。
- 如果你正在做 C，这一段会帮你少踩 D 这个坑。

### 正文结构

用二级标题分块，格式为“## 01、标题”、“## 02、标题”。

二级标题下可以用三级标题细分。三级标题格式为“### 三级xxx”。

默认结构不要少于这 6 块：
- 前言
- `## 01`
- `## 02`
- `## 03`
- `## 04`
- `## ending`

### Case 创意

要尽可能有趣，能让读者眼前一亮的案例。可以是实测数据、对比分析、独特的使用场景、开发者的真实反馈等。越具体越好，抽象的道理不如具体的故事。

如果涉及到coding，可以尝试和PaiAgent结合，这是一个Vibe Coding项目，源码在：https://github.com/itwanger/PaiAgent

也可以尝试和 paicoding.com 结合，看看能不能开发一些新的功能，实现一些新的业务，最好是和AI紧密结合，能写到简历上最好。

### 段落优先原则（重要）

正文内容优先使用段落式写法，用完整的句子和自然的过渡来表达观点。能用一段话说清楚的事情，就不要拆成列表。

只有在以下情况才使用列表：并列的技术栈或工具名称（比如技术栈包括 Spring Boot、MyBatis、Redis）、明确的操作步骤且步骤之间相互独立、需要强调的3个以上要点且确实是并列关系。

### 常用表达

自然融入这些口语化表达：“说真的”、“讲真”、“真心话”、“这一点至关重要”、“这个细节特别加分”、“好，我们直接来看效果”、“怎么样，是不是xxx？”，“啧啧啧”、“爽歪歪”（适度使用）。

### 文章结尾套路（重要）

结尾要给读者提供情绪价值，表达更深层的价值观和人生思考。这种段落能让读者从获取信息变成产生共鸣（不低于200个字）。

用短句和换行制造节奏感。不要写长段落，每句话单独一行或两三句一段。短促的节奏更有力量感。

用具体的生活场景代替抽象的道理。具体的画面比抽象的道理更打动人。

可以用对比制造张力，但避免连续二元排比和模板化口号。重点是具体场景里的真实反差，而不是套句式。

金句用加粗框【】起来。一段情绪升华最多一句金句，金句要短，要有记忆点。

可以往这些方向写：工作的意义不只是赚钱、技术是为了让生活更好、求职焦虑背后是对未来的期待、我们值得更好的工作环境、努力的人不应该被辜负、AI不是为了取代我们。

### 人性化表达规范

确保文章读起来像真人写的，避免 AI 生成的机械感。

**核心原则**

打破公式结构，避免二元对比和戏剧性分段。变化节奏，混合句子长度，两项优于三项。信任读者，直接陈述事实，跳过软化和辩解。

**句式和节奏**

长短句交替使用，不要连续出现结构相同的句子。比如不要连续三句都是“xxx是xxx”这种判断句。可以用反问、感叹、设问来调节节奏。段落结尾要多样化，不要每段都以总结句收尾。

**口语化表达**

适当加入口语词汇，比如“其实”、“说白了”、“讲真”、“反正”、“总之”、“话说回来”。这些词让文章更像聊天。

**个人视角**

多用“我”的视角来叙述，比如“我试了一下”、“我当时的想法是”、“我踩过这个坑”。避免全篇都是“用户可以”、“开发者需要”这种第三人称。

**必须避免的 AI 味词汇**

总结性套话：值得注意的是、需要指出的是、综上所述、由此可见、不难发现、此外、与此同时。

夸大意义的词：标志着、见证了、是……的体现/证明/提醒、凸显/强调/彰显了其重要性、为……奠定基础、不可磨灭的印记。

宣传性语言：充满活力的、丰富的（比喻）、深刻的、著名的、令人叹为观止的、开创性的、坐落于。

模糊归因：行业报告显示、观察者指出、专家认为、一些批评者认为、多个来源表明。

互联网黑话：赋能、抓手、闭环、打通、沉淀、对齐、拉通，除非是讽刺语境。

**必须避免的 AI 句式**

否定式排比：“不仅……而且……”、“这不仅仅是……而是……”被严重过度使用，直接删除。

三段式法则：不要强行把想法分成三组来显得全面，两项或四项更自然。

-ing 结尾的肤浅分析：删除“……，确保了……”、“……，体现了……”、“……，彰显了……”这类句尾。

过度限定：删除“可以说”、“在某种程度上”、“从某种意义上讲”这类软化词。

通用积极结论：删除“未来可期”、“前景光明”、“值得期待”这类空洞结尾。

**避免过度礼貌**

不要用“希望对您有所帮助”、“如有疑问请随时提出”这类客服式结尾。结尾要干脆，像朋友聊完天说“就这样，有问题评论区见”。

### 简历包装环节

如果文章涉及实战项目，可以加一个如何写到简历上的模块。

项目名称
项目简介：xxx

- xxxx 公式：用技术栈解决了什么问题、实现了哪些业务，有哪些量化数据

### 图片和流程图处理

文章中需要配图的地方，必须用占位符标注出来，方便后续插入。如果 ./sucai.md 中有相关截图，直接搬运过来即可。

如果 `sucai.md` 已经给了现成截图、现成图片链接或现成上一篇文章链接，优先复用，不要重新虚构一套无关截图。

### 截图占位符（强制）

终稿必须包含“截图占位符 + 关键信息关键词”，并满足：

- 每个核心章节（`## 01` 及之后）至少1个截图占位符。
- 每个占位符必须写清“截图目标”和“关键词”，方便后续检索与取证。
- 没有截图占位符的章节视为未完成，不允许交付。
- 如存在可核验的“跑分与外界评价”，该章节至少包含 1 个跑分截图占位符和 1 个 X 帖子截图占位符；如不存在，必须在文中明确说明缺失原因，不得杜撰。

**截图/图片占位格式（固定模板）：**

【此处插入<截图名称>：截图目标：<这张图要证明什么>；关键词：<关键词1>、<关键词2>、<关键词3>；建议位置：<命令行/网页/日志/IDE>】

示例：

【此处插入Claude Code 执行截图：截图目标：证明模型先拆解再执行；关键词：任务拆解、执行计划、变更说明；建议位置：终端会话窗口】

**流程图/架构图占位：** 如果需要流程图或架构图，用代码块包裹关键节点，比如：
```
用户输入 -> 意图识别 -> 知识库检索 -> LLM生成 -> 返回结果
```

### 互动钩子

在合适的地方加入互动引导，比如“还没有xxx的同学可以抓紧时间体验一波了”、“源码已经开源在GitHub上”、“订阅xxx的用户，可以xxx”。

## 禁止事项

不要用“首先、其次、最后”八股结构，这种写法太死板。不要过度使用emoji，偶尔用一两个可以。不要写超过5行的长段落，适时换行保持阅读节奏。不要用“让我们”、“我们来看看”这种翻译腔。不要只讲理论不给实操，读者要的是能上手的东西。不要在技术文章里加太多废话铺垫。
