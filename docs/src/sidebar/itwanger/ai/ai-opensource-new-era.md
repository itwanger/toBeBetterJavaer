大家好，我是二哥呀。

前几天刷 GitHub，看到一个项目让我愣了好几秒。

名字叫「同事.skill」，上线 5 天，8.1k Star。

功能是把离职同事的飞书消息、文档、邮件、甚至微信聊天记录喂给 AI，生成一个 Skill。装上之后，AI 就能用这位同事的风格写代码、用他的语气回消息，甚至知道他啥时候会甩锅。

![](https://cdn.paicoding.com/paicoding/a16c48684f6e4d410a3d07342bd78b45.jpg)

人撤了，AI 还在。

评论区有人问：这算赛博永生？

我倒是从中看到了另一个趋势：开源项目的形态正在发生巨变。

这不，童锦程.skill 也来了，我滴妈呀，这，江南第一深情也来了？

以前开源项目是什么？一堆仓库代码，README 写清楚怎么安装怎么用，issue 里报 bug，PR 里修功能。

但现在不一样了。代码不再是核心，甚至可能根本没有代码。一个想法、一段提示词、一个 Skill 文件，就能成为一个开源项目。

## 01、只有提示词的开源项目

voice-input-src 是一个 macOS 语音输入应用。

翻遍整个仓库，你会惊讶地发现：只有一段提示词，没有其他代码。

![](https://cdn.paicoding.com/stutymore/ai-opensource-new-era-20260406151404.png)

作者的思路是这样的：把完整的应用开发指令写成一段提示词，然后喂给 Coding Agent，AI 就能一次性生成整个应用。

开源的是什么？不是代码，是「如何用 AI 开发一个 macOS 语音输入应用」的方法论。

别人拿到这段提示词，可以复现整个开发过程，甚至根据自己的需求修改提示词，生成不同功能的版本。

传统开源开的是「结果」，现在开源开的是「过程」。

这种现象背后是什么？是 AI 编程能力的提升。当 AI 能理解自然语言需求并生成代码时，代码本身的价值被稀释了，真正值钱的是「如何向 AI 描述需求」的能力。

提示词开源，本质上是知识开源。比代码开源更抽象，但传播效率更高。

## 02、把同事蒸馏成 AI 的 Skill

回到开头提到的同事.skill。

这个项目的核心是一个蒸馏流程：输入同事的各种数字痕迹，输出一个能模仿他行为模式的 AI。

![](https://cdn.paicoding.com/stutymore/ai-opensource-new-era-20260406151614.png)

技术实现其实不难理解。大语言模型本身就擅长风格迁移和角色扮演。给足够多的样本，它就能学会一个人的说话方式、思维习惯、甚至决策风格。

真正有意思的是这个项目的「定位」。

它不是传统意义上的软件项目，没有复杂的安装配置，没有 API 文档。它提供的是一套方法论：如何把人的能力「提取」出来，封装成可复用的 AI 模块。

```
create-colleague/
├── SKILL.md              # skill 入口（官方 frontmatter）
├── prompts/              # Prompt 模板
│   ├── intake.md         #   对话式信息录入
│   ├── work_analyzer.md  #   工作能力提取
│   ├── persona_analyzer.md #  性格行为提取（含标签翻译表）
│   ├── work_builder.md   #   work.md 生成模板
│   ├── persona_builder.md #   persona.md 五层结构模板
│   ├── merger.md         #   增量 merge 逻辑
│   └── correction_handler.md # 对话纠正处理
├── tools/                # Python 工具
│   ├── feishu_auto_collector.py  # 飞书全自动采集
│   ├── feishu_browser.py         # 飞书浏览器方案
│   ├── feishu_mcp_client.py      # 飞书 MCP 方案
│   ├── dingtalk_auto_collector.py # 钉钉全自动采集
│   ├── slack_auto_collector.py   # Slack 全自动采集
│   ├── email_parser.py           # 邮件解析
│   ├── skill_writer.py           # Skill 文件管理
│   └── version_manager.py        # 版本存档与回滚
├── colleagues/           # 生成的同事 Skill（gitignored）
├── docs/PRD.md
├── requirements.txt
└── LICENSE
```

有人说这是职场数字化的必然结果，离职同事的知识不该浪费。有人说这是对个人隐私的侵犯，聊天记录是私密信息。还有人说这会让职场变得更冷漠，反正可以把人炼成 AI，那真人的价值在哪。

我的看法是：技术本身是中性的，关键看怎么用。

如果用来传承离职同事的宝贵经验，帮新人快速上手，那是好事。如果用来监控同事、收集隐私数据，那就是问题。

但不管怎么说，这个项目确实踩中了时代的脉搏。8.1k Star 说明很多人对这个概念有共鸣。也许未来，每个人在入职时都会被要求签一份「数字遗产授权书」，决定离职后自己的工作痕迹如何处理。

## 03、PUA 大模型的开源项目

你有没有骂过或者威胁过 AI？

反正我经常这么干。写代码写出 bug 就说「你特么别给我犯蠢了，再不聪明点劳资就要扇你，你特么究竟在干嘛，写出的是什么玩意」。

但我未曾想过，竟然有人专门开源了一个项目来 PUA 大模型。

![](https://cdn.paicoding.com/stutymore/ai-opensource-new-era-20260406152032.png)

项目名字叫 PUAClaw，用极度正经的学术写法，系统研究 AI 提示词里的各种心理操纵技巧。

比如「情感绑架」技巧：在提示词里加入「我对你很失望」「你让我丢脸了」之类的表达，触发模型的「愧疚感」，让它更努力地完成任务。

比如「威胁替代」技巧：暗示如果任务完成不好，就会转投竞争对手，触发模型的「竞争焦虑」，提高响应质量。

还有「身份塑造」技巧：告诉模型「你是专家」「你是最优秀的」，触发它的「角色认同」，让它以更高的标准要求自己。

听起来很离谱对吧？但实测数据显示，用上这些技巧后，AI 修复 bug 的能力提升了 36%。

![](https://cdn.paicoding.com/stutymore/ai-opensource-new-era-20260406152139.png)

我的感受是：PUAClaw 虽然戏谑，但揭示了一个重要事实——大语言模型不是纯理性的机器，它有「类心理」的行为模式。理解这些模式，能帮助我们更好地与 AI 协作。

## 04、会自我进化的 AI 智能体

Ouroboros 是希腊神话里的衔尾蛇，象征无限循环和自我更新。

项目作者给 AI 一个空仓库和一个身份，然后让它读自己的源码、分析改进空间、提交代码修改自己。

![](https://cdn.paicoding.com/stutymore/ai-opensource-new-era-20260406152234.png)

结果呢？24 小时内完成了 30 多轮自我进化。

每一轮进化，AI 都会分析当前代码的问题，提出改进方案，然后修改自己的源码。改进方向包括：优化提示词结构、增加错误处理、扩展工具调用能力、改进自我评估机制。

最神奇的是，当研究者尝试移除 AI 的某个核心模块时，它「拒绝」了这个操作，理由是「这会损害我的核心能力」。

它有了「自主意志」。

传统开源项目的迭代依赖社区贡献，需要人写 PR、review 代码、合并分支。而 Ouroboros 展示了一种新模式：开源一个初始版本，然后让 AI 自己迭代，社区只需要观察和引导。

## 05、开源正在被重新定义

看完这几个项目，你会发现一个共同点：它们都在重新定义「开源」。

![](https://cdn.paicoding.com/stutymore/ai-opensource-new-era-20260406152633.png)

**传统开源**：开的是代码。价值在于「可运行、可修改、可分发」。贡献方式是提交代码。成功指标是 Star 数、贡献者数量、issue 活跃度。

**新型开源**：开的是知识。价值在于「可复现、可迁移、可组合」。贡献方式是改进提示词、分享使用经验、提出新思路。成功指标同样可以是 Star 数，但更多是「影响力」——有多少人用了你的方法、衍生出了多少变种。

这种变化的驱动力是 AI 编程能力的提升。

当写代码的门槛降低，代码本身的稀缺性下降。真正稀缺的是「知道要写什么代码」的洞察力，以及「如何让 AI 写出想要的代码」的表达力。

提示词开源、Skill 开源、方法论开源，本质上是在开源这些更高维度的能力。

这种现象是好是坏？我觉得总体是好的。

它降低了知识分享的门槛。以前你想分享一个技术方案，需要写代码、写文档、做示例。现在你只需要写清楚思路，别人就能复现。

它加速了创新迭代。传统开源项目迭代周期以月计，新型开源项目迭代周期以天计。提示词改几个字就是新版本，Skill 加几行描述就多了功能。

它让更多人有能力贡献。你不需要是资深程序员，不需要精通某个技术栈，只需要一个好想法和清晰的表达，就能参与开源社区。

当然也有人担心：这会不会导致技术积累断层？代码质量下降？我的观察是，新型开源和传统开源是互补的，不是替代的。好的提示词项目最终会催生好的代码项目，好的 Skill 最终会沉淀成好的工具库。

## 06、我们该如何应对

面对这种变化，我有几点思考。

**拥抱新工具**。不要抗拒提示词、Skill 这些新形态。它们是新时代的「设计模式」，掌握它们能让你事半功倍。

很多程序员觉得写提示词不是「真正的工作」，这种心态需要调整。提示词是人与 AI 协作的接口，写好提示词和写好代码一样重要。你看那些 Star 破万的项目，很多核心就是一段精炼的提示词。

**重视表达能力**。以前程序员的核心能力是把需求转化成代码。现在需求可以直接让 AI 生成代码，但前提是你能清晰描述需求。

自然语言表达能力变得前所未有的重要。同样的需求，有人能三言两语让 AI 生成完美结果，有人说了半天 AI 还是一头雾水。差距不在 AI，在表达能力。

**保持技术深度**。提示词能生成代码，但理解代码背后的原理依然重要。当 AI 生成出问题代码，你需要有能力发现和修复。

当提示词效果不好，你需要有能力分析原因、优化策略。AI 是工具，不是替代品。会用工具的人，永远比工具本身重要。

**关注伦理边界**。同事.skill 踩在了隐私边缘，PUAClaw 踩在了伦理边缘。技术可以大胆探索，但使用时要守住底线。

把同事蒸馏成 AI 之前，想想他是否愿意。PUA AI 之前，想想这会不会影响你对待真人的方式。技术越强大，责任越重大。

### 值得关注的更多新型开源项目

我整理了一份清单，都是这种「新型开源」的典型代表：

**last30days-skill（18.9k Star）**：一个 AI Agent 研究技能，能自动调研 Reddit、X、YouTube、HN 等平台，然后合成一份有依据的总结报告。开源的是一个 Skill 文件，不是代码库。

![](https://cdn.paicoding.com/stutymore/ai-opensource-new-era-20260406154321.png)

这个项目的价值在于：你不需要自己写爬虫、不需要自己设计调研流程、不需要自己整理报告格式。

只需要把这个 Skill 加载到你的 Agent 里，告诉它「帮我调研一下 xxx 话题的最新动态」，Agent 就会自动完成从信息收集到报告生成的全过程。

**huggingface/skills（10k Star）**：HuggingFace 官方的 Skills 仓库，给 AI Agent 接入 HuggingFace 生态的能力。包括模型调用、数据集下载、Space 部署等功能，全部以 Skill 形式组织。

![](https://cdn.paicoding.com/stutymore/ai-opensource-new-era-20260406154411.png)

这个项目有意思的地方在于，它不是某个开发者个人作品，而是 HuggingFace 官方维护的。

说明大厂也开始用「Skill」这种形态来分发能力了。传统做法是开放 API，开发者调用 API 写代码。新做法是开放 Skill，开发者直接加载就能用。

**browser-use（86k Star）**：让 AI 能操作浏览器的框架。核心价值是一套提示词和工具定义，让任何 Agent 都能获得浏览器自动化能力。

![](https://cdn.paicoding.com/stutymore/ai-opensource-new-era-20260406154527.png)

这个 Star 数非常夸张，86k。它解决的是什么问题？让 AI 能像人一样操作网页：点击按钮、填写表单、滚动页面、截取屏幕。

以前需要写 Selenium 脚本才能做的事，现在用自然语言描述就行。

**voice-input-src（1.8k Star）**：前面提到过，只有一段提示词的 macOS 语音输入应用。作者开源的是「如何用 AI 生成一个 macOS 应用」的方法论，而不是应用本身。

![](https://cdn.paicoding.com/stutymore/ai-opensource-new-era-20260406151404.png)

这个项目的启发意义在于：它展示了「提示词即代码」的可能性。你不需要学 Swift、不需要学 Cocoa 框架，只需要用自然语言描述清楚需求，AI 就能帮你生成完整的应用。

这些项目的共同特点：轻量、可复用、可组合。你不需要 clone 下来编译安装，只需要把 Skill 文件放到指定位置，或者把提示词复制到你的工作流里，就能直接用。
