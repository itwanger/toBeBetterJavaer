老王开门见山地问："OpenClaw的memory了解吗？"

我和老王四目相对那一刻，我懂他想要的答案："必须啊，伪无限记忆，语义检索，拿捏。"

OpenClaw对memory的定义是持久存储在磁盘上的结构化信息。

只要有足够的存储空间，记忆就可以无限增长，能够跨会话保留，在使用时按需检索，且存储成本几乎为零。

更为关键的是，这种记忆能够支持语义检索，也就意味着不需要把所有历史信息都塞进上下文，只需要检索当前任务相关的片段。

"逗逗你的呀，别那么上头。"老王摸了摸他的光头，捋了捋他的胡子，"那我问你：OpenClaw的memory架构是怎样的？别给我整概念，讲细节。"

## content

### 01、memory的两层架构是什么？

"王哥，你这个问题问得好。很多人以为memory就是一个文件，错。"

OpenClaw的memory采用的是两层架构：静态记忆和动态记忆。

【此处插入memory架构图：截图目标：展示memory的两层架构；关键词：静态记忆、动态记忆、MEMORY.md、JSONL；建议位置：架构示意图】

#### 静态记忆：Markdown文件

静态记忆存储在`~/.openclaw/workspace/MEMORY.md`和`memory/*.md`文件中，可以手动创建，也可以自动生成。

这是整个系统的长期记忆，可以理解成是在短期的琐碎记忆中提炼出来的需要系统重点记住的内容，比如用户的性格、身份信息、回答偏好等。

举个例子，你告诉Agent："我是Java后端开发，回答问题时请用Java相关技术栈。"这句话就会被提炼成静态记忆，存储在Markdown文件中。下次对话时，Agent会自动检索到这条记忆，按照你的偏好回答。

#### 动态记忆：JSONL文件

动态记忆存储在`~/.openclaw/agents/{agentId}/sessions/*.jsonl`文件中，自动记录。

每次用户与Agent交互时，系统会自动将对话内容追加到JSONL格式的会话日志文件中，这是最原始的、未经处理过的记忆。

老王追问："那静态记忆和动态记忆之间是怎么转换的？总不能一直往JSONL里追加吧？"

我说："王哥，问到点子上了。"

### 02、记忆是如何自动转换的？

"记忆转换有两种触发机制。"

#### 机制一：session-memory Hook

当用户执行`/new`命令重置会话时，系统会触发session-memory Hook，自动将上一个会话的关键内容转换为Markdown文件。

这个过程是自动化的，你不需要手动干预。系统会分析JSONL文件中的对话内容，提取出关键信息，比如用户的偏好、重要的上下文、需要长期记住的事实等，然后写入`memory/YYYY-MM-DD.md`文件。

#### 机制二：Memory Flush

这是一个非常关键的自动化机制。

当会话上下文接近token限制时，系统会在压缩（compaction）前触发一个特殊的Agent回合，在该回合中，Agent被明确指示：将需要持久保存的重要信息写入`memory/YYYY-MM-DD.md`文件。

这就像人的大脑在睡眠时整理白天的记忆一样，把重要的东西存下来，不重要的丢弃。

老王点点头："那这些Markdown文件是怎么被检索的？总不能每次都遍历所有文件吧？"

### 03、memory 的索引到底是怎么建起来的？

"王哥，真正难的不是‘记下来’，而是‘下次还能在几百份记忆里把它找回来’。"

【此处插入语义检索流程图：截图目标：展示索引构建和检索流程；关键词：Markdown、SQLite、sqlite-vec、FTS5；建议位置：流程示意图】

OpenClaw 在 memory 这件事上，其实分得很清楚：

- **Markdown 文件是记忆本体**，也就是 source of truth。
- **SQLite 索引只是加速层**，负责把这些 Markdown 变成“可检索”的东西。

这句话很重要。

很多系统一上来就把“数据库”当 memory 本体，最后用户连自己的记忆存在哪都说不清。

OpenClaw 反过来：记忆先落成普通文件，用户能看、能改、能 git 管；检索只是为了让 Agent 别在几百个 Markdown 里徒手翻页。

#### 先有文件，后有索引

从官方文档看，默认会参与 memory 的，是这两层 Markdown：

- `MEMORY.md`：长期记忆，偏“结论”和“偏好”
- `memory/YYYY-MM-DD.md`：日记式记录，偏“当天发生了什么”

而 `memory_search` 建的索引，默认就是围绕这些 Markdown 文件展开的。

如果从命令行看会更直观：

```bash
openclaw memory status
```

我这台机器当前看到的是：

```bash
Provider: openai (requested: openai)
Model: nomic-embed-text
Store: ~/.openclaw/memory/paismart.sqlite
Indexed: 8/8 files · 14 chunks
Vector: ready
FTS: ready
```

这几行字其实已经把底牌掀开了。

它说明 OpenClaw 不是“每次查询时临时扫目录”，而是提前把 Markdown 切块、建索引、落到一个 SQLite 文件里。等 Agent 真要查历史时，直接查索引就行。

#### 建索引时到底做了什么？

可以把它理解成四步。

**第一步，发现变更。**

OpenClaw 会监控 `MEMORY.md` 和 `memory/*.md` 的变化。新增了文件，或者文件内容有更新，就把这个文件标记成 dirty，准备重新建索引。

**第二步，切块。**

一整篇 Markdown 不会当成一个大对象直接塞进向量库。

原因很简单：太大。

如果一整篇都做成一个 embedding，那么一个文件里既有“二哥的沟通偏好”，又有“网络配置”，再加上“文章写作风格”，最后得到的就是一个什么都沾一点、但什么都不够准的平均向量。

所以更合理的做法是切成多个 chunk，让“一个块只表达一小段相对完整的意思”。

【此处插入文本分块示意图：截图目标：展示分块原理；关键词：chunk、切块、边界、语义完整性；建议位置：示意图】

**第三步，双路建索引。**

每个 chunk 不只会走一遍 embedding，还会同时走全文检索索引：

- 一路进入 **向量索引**，负责“意思差不多也能搜出来”
- 一路进入 **FTS 全文索引**，负责“关键词命中要准”

也就是说，OpenClaw 不是只做向量检索，也不是只做关键词检索，而是把两套检索能力都准备好。

**第四步，落到本地 SQLite。**

最后这些 chunk、元信息、全文索引、向量索引，都收进一个本地 SQLite store 里。

这也是 OpenClaw memory 最有工程味的一点：

它没有把一个“个人助理的记忆系统”做成需要单独部署 Milvus、ES、Redis、消息队列、后台任务框架的企业级怪兽。

它更像一个讲道理的本地工具：**文件做真相，SQLite 做加速，能跑起来比架构图漂亮更重要。**

老王点点头："那检索时到底怎么查？是纯向量，还是纯关键词？"

我说："都不是，它玩的是混合检索。"

### 04、混合检索为什么比纯向量更靠谱？

"王哥，纯向量检索这几年被吹得有点像万能药，但真到工程里，很多时候它也会犯迷糊。"

【此处插入混合检索示意图：截图目标：展示混合检索原理；关键词：FTS5、BM25、sqlite-vec、rerank；建议位置：示意图】

举个最简单的例子。

如果我们搜的是：`memory_search("nomic-embed-text")`

这类查询的关键，不是“语义接近”，而是“这个字符串必须命中”。

如果只靠向量检索，它可能把“embedding 模型”“本地向量索引”“OpenAI provider”这些语义邻居都捞出来，但偏偏把最关键的那一段排得不够靠前。

可如果搜的是：

> 上次说过的那个文章写作偏好是什么来着？

这时候关键词检索又不够用了，因为用户未必会原样说出“娓娓道来”“少用你、多用大家和我们”这些固定字眼。

所以 OpenClaw 的思路很朴素：

- **FTS5 + BM25** 负责精确词项命中
- **sqlite-vec** 负责语义相似召回
- 最后再把两边的结果做融合，返回更像“人类记忆召回”的结果

#### 为什么是 FTS5？

因为 SQLite 的 FTS5，本质上就是一个轻量级全文搜索引擎。

它比 `LIKE '%xxx%'` 强的地方，不只是快，而是它知道“哪些词更重要，哪些结果应该排前面”。

BM25 的价值就在这。

它不会傻乎乎地认为一个词出现 10 次就一定比出现 2 次更重要，而是会结合：

- 词频
- 文档长度
- 这个词在整个语料里稀不稀有

于是像 `memory_flush`、`session-memory`、`nomic-embed-text` 这种比较稀缺的词，相关性权重天然就更高。

#### 为什么还要 sqlite-vec？

因为 FTS5 再强，也主要解决“字面命中”的问题。

可真实对话不是搜索框。

用户经常会说：

- “上次那个事”
- “之前你记住的偏好”
- “我不是说过不要那种爆款腔吗”

这种问法，字面上未必能正好撞到原文，但语义是接近的。

这时候 embedding 的价值就出来了。

它先把 query 编成向量，再和每个 chunk 的向量做近邻比较，把语义接近的片段拉出来。

可以粗暴理解成：

```text
query
  ↓
embedding(query)
  ↓
和 chunks_vec 里的每个向量算距离
  ↓
取 top-k
```

这套东西要是放到很多 SaaS 产品里，下一步一般就要开始祭出一个单独的向量数据库了。

但 OpenClaw 没这么干。

它直接用 `sqlite-vec` 这类 SQLite 扩展，把向量检索能力放进本地 SQLite 里。

这背后的取舍很明显：

- 个人场景、几百到几千段 memory，没必要上重武器
- 本地优先，单文件存储更容易迁移、备份、调试
- 出问题时，定位范围小，不会变成“到底是 embedding 服务挂了，还是向量库挂了，还是同步任务没跑”

说白了，这不是在追求“最先进”，而是在追求“对个人 Agent 足够先进”。

#### 混合检索的真正意义

很多人一听“混合检索”，第一反应是技术栈堆料。

其实不是。

它真正解决的是两类问题：

**第一类，专有名词问题。**

像 `openclaw memory status`、`memory_search`、`session-memory`、`NO_REPLY`，这种词必须靠关键词保精度。

**第二类，自然语言问题。**

像“上次那个偏好”“之前说过别做爆款腔”，这种表述必须靠向量保召回。

把这两边拼起来，才比较接近人类说的“记得”。

老王听到这儿笑了："行，概念算你讲明白了。那 Agent 自己到底怎么用这些记忆？别又停在存储层。"

### 05、检索到记忆之后，Agent 是怎么把它用起来的？

"王哥，这一步才是 memory 真正发生价值的地方。"

很多人以为 memory 系统的终点是“查到了”。

其实不是。

终点是：**查到的那几段，最后有没有变成当前这次回答的一部分。**

OpenClaw 主要给 Agent 暴露了两个工具：

#### 1）`memory_search`

这是“先回忆一下”的入口。

当 Agent 发现问题涉及过去的决策、偏好、历史上下文，它不会把整个 `memory/` 目录读一遍，而是先发起一次语义搜索。

例如：

```text
memory_search("二哥的文章写作偏好")
```

返回的不是整篇文件，而是**最相关的若干 snippet + 文件路径 + 行号范围**。

这样做有两个好处：

- 控制 token，别把整个历史一口气塞进上下文
- 先粗召回，找到“值得展开读”的位置

#### 2）`memory_get`

这是“命中了，再精读”的入口。

如果 `memory_search` 返回说，关键信息在：

- `MEMORY.md#L1-L16`
- `memory/2026-03-19.md#L20-L48`

那 Agent 下一步就可以用 `memory_get` 去读具体文件的具体行段，而不是把全文件都搬进来。

这个设计其实很像现代代码检索：

- 先搜索
- 再跳转
- 最后按需展开上下文

#### 一次完整调用链，长什么样？

如果把它画成链路，大概是这样：

```text
用户提问
  ↓
Agent 判断：这事和过去有关
  ↓
调用 memory_search(query)
  ↓
返回 top-k snippets
  ↓
必要时调用 memory_get(path, from, lines)
  ↓
把取回的片段并入当前推理上下文
  ↓
生成回复 / 执行动作
```

注意，这里面有个特别容易被忽略的点：

**memory 文件本身不是每回合全量注入。**

官方文档明确写了，`memory/*.md` 这种 daily 文件默认并不会自动塞进上下文窗口，而是通过 `memory_search` 和 `memory_get` 按需读取。

这就解释了为什么 OpenClaw 的 memory 能够“越记越多”，但又不至于每轮对话都把上下文撑爆。

也就是说，它追求的不是“全都记在 prompt 里”，而是：

> 该常驻的常驻，该检索的检索。

像 `AGENTS.md`、`USER.md`、`SOUL.md` 这类“每回合都必须知道”的文件，会直接注入。

但 daily memory 这种“可能有用，也可能今天根本用不上”的内容，就延迟到真的需要时再读。

#### memory flush 为什么是这个体系里的关键一环？

因为如果只会检索，不会在上下文爆掉之前把重要信息写回磁盘，那前面做得再漂亮，也只是“短期聪明”。

OpenClaw 在会话接近 compaction 之前，会触发一次 **silent memory flush**。

官方文档的描述很直接：当 session 接近自动压缩时，系统会发起一个静默回合，提醒模型把值得长期保留的内容写进 `memory/YYYY-MM-DD.md`。

这一步特别像人在睡前记日记。

不是把白天每句话都誊抄一遍，而是把“以后可能还会用到的东西”留下来。

于是整个 memory 闭环就完整了：

```text
对话发生
  ↓
重要信息写入 Markdown
  ↓
后台更新 SQLite 索引
  ↓
后续问题通过 memory_search 召回
  ↓
命中片段通过 memory_get 精读
  ↓
重新进入当前推理
```

老王听到这儿，总算不再追问了。

因为讲到这里，memory 就不再只是一个“能记事的小功能”，而是一条完整的工程链路了。

### 06、memory使用中的坑和最佳实践

"王哥，这个问题太实际了，我踩过不少坑。"

#### 坑一：记忆文件过大

如果记忆文件太大，索引构建会很慢，检索也会变慢。

解决方法：定期清理不需要的记忆，或者把记忆文件按主题拆分。

#### 坑二：Embedding模型和语料不匹配

检索效果差，未必是 memory 机制有问题，很多时候是 embedding 模型和你的语料不匹配。

比如技术文档、命令行、配置项、英文术语很多的场景，如果模型对这类文本不敏感，召回结果就容易发散。

解决方法也别太教条：

- 先看语料类型，再选 embedding 模型
- 本地优先的场景，可以先试像 `nomic-embed-text` 这样的本地/自托管方案
- 对召回质量要求特别高时，再评估远程 embedding 服务

#### 坑三：检索结果不相关

有时候检索出来的结果和查询不太相关，尤其是查询语句比较模糊的时候。

解决方法：优化查询语句，或者在查询前先对问题进行分析和改写。

#### 最佳实践

我总结了几个最佳实践：

- 定期整理MEMORY.md，保持内容简洁
- 对于重要的偏好，手动写入MEMORY.md，不要依赖自动提取
- 使用OpenAI的Embedding服务，检索效果更好
- 在 `AGENTS.md`、`SOUL.md`、`USER.md` 里把 memory 的使用边界和习惯写清楚

### 07、实战：让Agent记住我的偏好

"王哥，我给你讲一个真实的场景。"

我有一个Agent专门帮我审核gitcode账号。每次审核的时候，我都要告诉它一些重复的信息，比如：

- 审核完成后发消息到哪个飞书群
- 添加到哪个gitcode项目组
- 审核结果用什么格式回复

这些信息每次都要重复说，很烦。

后来我把这些偏好写进了MEMORY.md：

```markdown
# 用户偏好

## gitcode审核
- 审核完成后发消息到"技术派-运营群"
- 添加到项目组：技术派-会员组
- 回复格式：@用户 审核通过，已添加到技术派-会员组

## 其他偏好
- 我是Java后端开发，回答问题请用Java技术栈
- 回复时请简洁，不要废话
```

现在每次审核，Agent都会自动检索这些偏好，按照我的要求执行，再也不用重复说了。

老王听完眼睛一亮："这个场景实用，省得每次都重复说一堆废话。"

我说："还不止。我还让Agent记住了我的工作习惯。比如我喜欢在早上处理审核任务，Agent会在每天早上主动提醒我有多少待审核的申请。"

### 08、memory的未来演进方向

老王最后问了一个开放性的问题："你觉得OpenClaw的memory未来会怎么演进？"

我说："王哥，我觉得有几个方向。"

**方向一：更智能的记忆提取**

目前的记忆提取是基于规则的，未来可以用更强的模型来做记忆提取，自动识别哪些信息值得长期记住。

**方向二：记忆的层次化组织**

目前的记忆是扁平的，未来可以引入层次化的组织方式，比如按主题、按时间、按重要性分类。

**方向三：跨Agent记忆共享**

目前每个Agent有自己独立的记忆，未来可以实现跨Agent的记忆共享，让多个Agent共享同一套知识库。

**方向四：记忆的遗忘机制**

目前的记忆只增不减，未来可以引入遗忘机制，自动清理过时或不再相关的记忆。
