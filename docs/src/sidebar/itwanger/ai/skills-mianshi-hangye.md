老王是某大厂的一位高级工程师，一头秀发直垂肩背，人不谈不上英俊，但气宇不凡。

他的第一个问题是：“你简历上写的熟悉 Agent Skills，那你说说 Skills 和 Prompt 到底有什么区别？”

![](https://files.mdnice.com/user/3903/073aca12-a4f0-413c-a848-c5232eb4ed7c.jpg)

瞅着老王的秀发出了神，以至于那一刻我脑子一片空白，支支吾吾说了句“Skills 就是封装好的 Prompt”。

话音刚落，回过神的我明显感觉老王不是很满意。

但老王很有耐心，没有 KPI 我，仍然给我聊了 35 分钟 Skills 的本质。

说真心话，遇到这样的面试官，真的打心眼里佩服，不会因为一时的得失就挂你，反而会引导你，让你敞开心扉地表达自己。

感恩。

## 01、Skills 和 Prompt 的本质区别

**老王：** 能具体说说吗？

好的，王哥。Prompt 是我们和 AI 的一次临时沟通，这次说完，这次生效。下次还想复用同样的能力，大概率还得再说一遍，或者去聊天记录里考古，复制粘贴回来。

![](https://files.mdnice.com/user/3903/bb59c4c9-0a59-4516-a06e-6f7d5e07c1e6.png)

Skills 不一样。它是把一类问题的解法沉淀下来，变成一个可以反复调用的模块。就像 Java 的宣传语“一次编写，到处运行”。

**老王：** 那不就是 Prompt 的模板化吗？

王哥，要只是模板化，那 Skills 也太委屈了。模板化偏静态，Skills 则是动态的、能持续进化的。

举个例子。假设我们经常让 AI 帮做代码审查。用 Prompt 的方式，每次都得重新交代一遍：“帮我审查这段代码，重点关注安全性、性能、可读性，按阿里巴巴 Java 规范来。”

用 Skills 的方式，就是把这套审查逻辑封装成一个 Skill。下次直接说“用 code-review Skill”，AI 基本就知道该做什么、怎么做、做到什么程度。

更重要的是，这个 Skill 还能持续升级。发现漏掉了并发安全检查，就补进去；团队规范更新了，就同步更新。所有在用这个 Skill 的人，都能立刻吃到这波“版本红利”。

![](https://files.mdnice.com/user/3903/c378abd1-0489-43ae-a2ee-a1a512d64399.png)

**老王：** 有点意思。那你觉得 Skills 的核心价值是什么？

王哥，我会用三个词来概括：沉淀、复用、进化。

沉淀是把个人经验变成团队资产。复用是让好的实践快速传播。进化是让能力持续迭代升级。

这三点，Prompt 不是完全做不到，但做起来很吃力，也很容易散。

**老王：** 能举个例子说明“进化”吗？

当然。我们团队早期有一个代码审查的 Prompt，大概是这样的：“帮我审查这段代码，关注安全性和性能。”

用了一段时间后，我们发现总是漏掉并发安全问题。于是 Prompt 改成了：“帮我审查这段代码，关注安全性（包括并发安全）、性能、代码可读性。”

又过了一段时间，团队规范更新了，要求所有方法必须有 JavaDoc 注释。Prompt 又得改。

问题就在这儿。这个 Prompt 很可能散落在不同同事的笔记里、聊天记录里、飞书收藏夹里。你改了你的版本，别人可能还在用上个月的旧版本。最后团队执行标准就很难统一。

用 Skills 就不一样了。把审查逻辑封装成 Skill，发现漏掉并发安全，就更新 Skill；规范更新了，就同步更新 Skill。所有用这个 Skill 的人，立刻就能用上最新版本。

这就是进化能力。Prompt 是静态的，Skills 是动态的、可迭代的。

## 02、Skills 在团队协作中的价值

**老王：** 你刚才提到了团队资产，能展开说说 Skills 在团队协作中的价值吗？

王哥，这恰恰是我觉得 Skills 最值钱的地方。

传统的团队协作，很多时候还是“老师傅带徒弟，靠嘴传嘴”。一个资深工程师积累了一套代码审查经验，新人来了，他得一对一地讲，讲完对方还未必一次就能记住。

有了 Skills，这套经验就能被封装成一个 Skill。新人来了，先用起来再理解也没问题。资深工程师的经验，会以 Skills 的形式在团队里快速扩散。

**老王：** 但这和文档有什么区别？写个代码审查规范文档不也一样吗？

文档是写给人看的，Skills 是写给 AI 执行的。这是两者最本质的区别。

![](https://files.mdnice.com/user/3903/69c2a621-abea-4580-baf5-8c7b6acdbaf7.jpg)

人看文档，要经历理解、记忆、应用这三关。理解可能有偏差，记忆可能有遗漏，应用时还可能顺手加点“个人发挥”。最后每个人执行出来的效果，往往就不太一样。

Skills 直接给 AI 用，执行口径会统一得多。你定义了什么边界、什么流程、什么标准，它就更容易按这套来，不容易跑偏。

**老王：** 那团队里每个人都用同一个 Skill，会不会导致思维僵化？

恰恰相反，Skills 让团队有更多精力去做创造性工作。

是这样，代码审查这种重复性工作，本来就不太需要每个人都来一段“个人艺术创作”。规范就是规范，按成熟实践来，反而更稳。

但 Skills 节省下来的时间，可以用来做真正有创造性的事情。比如设计更好的架构、解决更复杂的技术难题。

而且 Skills 本身也是可以迭代的。团队在实践中发现更好的做法，可以更新 Skill，所有人立刻受益。这是一种良性的进化机制。

**老王：** 你们小组现在有多少个 Skills？

王哥，我们小组这半年下来，沉淀了 20 多个我觉得比较精华的 Skills，覆盖代码审查、文档生成、测试用例设计、技术方案评审等场景。

最常用的是代码审查 Skill。以前人工审查一个 PR，平均要 60 分钟；现在用 Skill 辅助，30 分钟左右就能完成，而且遗漏的问题还更少。

**老王：** 30 分钟是怎么做到的？

分两步。第一步，AI 用 Skill 自动扫描代码，找出潜在问题。这一步大概 10 分钟。第二步，人工 review AI 的扫描结果，确认问题、补充上下文。这一步大概 20 分钟。

以前人工审查，要从头到尾读代码、理逻辑、找问题，流程挺费脑子。现在 AI 先帮你把潜在问题圈出来，你只需要判断“这是不是问题”“严重程度怎么样”。

![](https://files.mdnice.com/user/3903/2e05c96f-afb2-4090-aa5a-137d8a036be3.png)

而且 AI 不会疲劳。人 review 到第 5 个 PR 的时候，注意力多少会有点飘，漏问题很正常；AI 不一样，它至少不会因为午饭吃多了就降低标准。

**老王：** 那 AI 会不会误报？

会，但比例不高。大概 2% 的 AI 提示是误报。但即使这样，也比人工审查遗漏问题要好。

而且随着 Skill 迭代，误报率还会继续往下走。我们发现某类误报比较多，就在 Skill 里补规则、加约束。这本身就是一个持续打磨的过程。

## 03、Skills 的工作原理

**老王：** 你刚才说 Skills 是动态加载的，能解释一下原理吗？

好的，这里就得聊到 AI 的上下文窗口限制了。

![](https://files.mdnice.com/user/3903/b49039fd-2093-44be-843c-c5c2f5535893.png)

我们知道，AI 的上下文窗口是有限的。比如 Claude 是 200K。你要是把所有 Skills 一股脑全塞进去，窗口很快就满了，AI 还没开始干活，粮草先见底了。

所以，Skills 才会采用一种叫“渐进式披露”的架构。

**老王：** 渐进式披露？

**王哥：** 对。简单说，不是让 AI 一上来知道得越多越好，而是让它在恰当的时间，知道恰当的事。

拆开看，其实就是三步：

第一步，发现阶段。AI 启动时，只加载所有 Skills 的元数据，比如名称、描述、触发关键词。这就像先翻图书馆目录卡片，先知道哪本书大概讲什么，不会一上来就把整个书架扛回家。

第二步，语义匹配。当用户输入问题，AI 会根据语义去匹配相关 Skills。比如你问“帮我审查这段代码”，它脑子里就会先把“代码审查 Skill”拎出来看看，是不是这题的标准答案。

第三步，执行阶段。匹配成功后，才会加载这个 Skill 的完整内容。这时候 AI 才真正“看到”详细指令和约束，相当于终于拿到了完整版操作手册，可以开始认真干活了。

![](https://files.mdnice.com/user/3903/1357c7ef-6183-4ab3-965e-a0fff3b1e6f0.jpg)

**老王：** 这个匹配过程是怎么实现的？

本质上，它背后跑的是向量相似度计算。

每个 Skill 的元数据会被转换成向量，用户输入的问题也会被转换成向量。然后一算相似度，分高的那些，就是更可能匹配上的 Skill。

这个过程非常快，用户基本感知不到延迟。也正因为这样，只有匹配到的 Skill 才需要加载完整内容，其他 Skills 先在旁边待命，只占一点点元数据空间。

这也就是为什么你明明挂了几十个、甚至上百个 Skills，却不用天天担心上下文“啪”一下炸掉。

**老王：** 你说 Skills 可以迭代进化，那创建一个 Skill 的过程是怎样的？


![](https://files.mdnice.com/user/3903/8337edcd-b91e-4e9d-a004-da2d7328019e.png)


王哥，我正好拆解过 Claude Code 内置的 create-skill 源码。它整个创建流程，说白了可以拆成 6 步：

**第一步，Understanding**——先搞清楚你到底想解决什么问题。系统会主动追问你：这个 Skill 解决什么问题？有哪些使用场景？触发条件是什么？这一步特别像产品经理追需求。很多人一上来就开写，最后写出来的东西，字不少、结构也挺完整，就是不解决真问题。

**第二步，Planning**——把需求拆解成可复用的资源类型。哪些功能需要脚本实现，就放 `scripts/`；哪些内容更适合作为参考文档，就放 `references/`。别看名字朴素，这一步本质上已经是在做一次轻量架构设计了。

**第三步，Initializing**——调用 `init_skill.py` 脚本，自动生成标准目录结构和模板文件。该留空的地方会留空，该提醒你的地方也会老老实实标上 TODO，省得你还没开始写内容，先把目录结构写乱了。

**第四步，Editing**——填充具体内容。这是最耗时的环节，也是最见功力的地方。description 必须把触发条件写清楚，`SKILL.md` 也不能一路写成小说，最好控制在 500 行以内。

**第五步，Packaging**——调用 `package_skill.py` 打包。打包前还要过六重验证：文件存在性、Frontmatter 格式、YAML 解析、必填字段、命名规范（必须是 hyphen-case）、长度限制（description 不超过 1024 字符）。这几关全过了，才能顺顺当当地打包成 `.skill` 文件。

**第六步，Iteration**——根据实际使用反馈持续优化。

**老王：** 六重验证，这个设计挺严格的。

是的，就像代码提交前的 CI 检查，专门在最后一道门口拦你一下，防止把半成品混出去。而且整个流程有个很妙的设计，叫 **显式复杂、隐式简单**。底层其实涉及 3 个 Python 脚本、数十个函数调用，但用户看到的体验却是顺滑的、引导式的。你不用把底层机制全背下来，跟着提示一步步走，就能把事做成。


## 04、Skills 与 MCP 的关系

**老王：** 去年，MCP 很火，你觉得 Skills 和 MCP 是什么关系？

我觉得它们更像一对搭子，属于谁也替代不了谁的那种，是互补关系，不是竞争关系。

MCP（Model Context Protocol）解决的是 AI 和外部系统怎么连起来的问题。比如让 AI 调 GitHub API、查数据库、操作文件系统、打开浏览器。说白了，就是先把 AI 的手脚接上。

我就经常用 Chrome 的 Devtools MCP 让 AI 进行自动化测试。

![](https://files.mdnice.com/user/3903/cb5ef27d-aeb2-4c62-9c72-20244af02eb8.png)

Skills 解决的，则是 AI 的能力怎么封装、怎么复用的问题。比如让 AI 具备代码审查、文档生成、技术方案评审这些能力。说白了，就是再把它做事的方法教明白。

**老王：** 能打个比方吗？

可以。MCP 像是给 AI 装上了机械臂，让它终于能摸到外部世界；Skills 则像是给了它一套操作手册，让它知道该怎么把事做对。

只给 MCP，不给 Skills，AI 可能会变成一个“工具很多但不太会用”的实习生；只给 Skills，不给 MCP，它又像一个道理都懂、但碰不到真实系统的军师。

两者结合起来，AI 才更像一个能真正下场干活的生产力工具。

**老王：** 那未来会是一个什么样的格局？

我觉得会有两个趋势，而且都挺现实。

第一个趋势是 Skills 标准化。现在每个团队的 Skills 格式可能都不一样，你家一套写法、我家一套目录，彼此看着都像“方言”。未来大概率会慢慢收敛出行业标准，就像 npm、pip 那样。

第二个趋势是 Skills 商店化。个人开发者可以把 Skills 发布到商店，其他人付费或免费使用。真走到那一步，新的生态基本就起来了，甚至会冒出一批“靠卖好 Skill 吃饭”的人。

![](https://files.mdnice.com/user/3903/f4e0b634-eda4-4500-a439-c899beda483b.jpg)

**老王：** 这个观点很有意思。那你觉得未来程序员的核心竞争力是什么？

**王哥：** 我觉得最后还是会落到三个能力上。

第一，问题定义能力。AI 能帮我们解决问题，但前提是你得先把问题定义清楚。很多人连自己到底要什么都说不明白，AI 再强，也只能陪着一起原地转圈。

第二，系统设计能力。AI 能写代码，但架构设计、模块划分、接口定义，这些还是得人来把控。AI 更像执行者，人更像设计者。方向盘，暂时还得握在人手里。

第三，持续学习能力。技术更新这么快，今天有用的 Skills，明天可能就过时了。保持学习，才能保持竞争力。

Skills 恰恰能帮助程序员把这三个能力往上抬一截。它帮我们省下时间，把精力腾出来，去思考更关键的问题定义和系统设计。很多时候，真正拉开差距的，不是你打字有多快，而是你想问题有多准。

## 05、如何设计一个好的 Skill

**老王：** 假设我现在要你设计一个“技术方案评审”的 Skill，你会怎么设计？

首先，先把这个 Skill 的目标钉死：帮助团队快速评审技术方案，发现潜在风险，给出改进建议。

然后，再把评审维度拆出来：

- 可行性：技术方案是否可行，有没有技术难点
- 性能：是否满足性能要求，有没有瓶颈
- 安全：有没有安全风险，数据保护是否到位
- 可维护性：代码是否易于维护，文档是否清晰
- 成本：实现成本是否合理，有没有更优方案

**老王：** 这些维度会不会太通用？

有一点，所以还得结合具体业务做定制。比如金融系统要格外盯安全，恨不得拿放大镜看；电商系统就得把性能和并发放到更靠前的位置，毕竟高峰期一来，系统可不会跟你讲情怀。

接下来，再把输出格式钉住：

```
## 方案概述
[一句话概括这个方案]

## 评审结果
- 可行性：✅/⚠️/❌ [说明]
- 性能：✅/⚠️/❌ [说明]
- 安全：✅/⚠️/❌ [说明]
- 可维护性：✅/⚠️/❌ [说明]
- 成本：✅/⚠️/❌ [说明]

## 主要风险
1. [风险描述]
2. [风险描述]

## 改进建议
1. [建议]
2. [建议]

## 总体评价
[优秀/良好/需改进]
```

![](https://files.mdnice.com/user/3903/be502cec-f72f-44ae-92f4-e752930ce7f6.jpg)

**老王：** 这个 Skill 怎么使用？

在对话里直接输入：“用 tech-review Skill 评审这个方案：[粘贴方案内容]”

AI 会自动加载 Skill，然后按预设的维度和格式，把评审结果规规矩矩地吐出来。你不用每次再从头提醒它“记得看风险、记得看性能、记得给建议”。

**老王：** 如果评审结果不满意怎么办？

那就继续迭代 Skill。比如发现总是漏掉某个风险点，就把对应检查项补进去；发现输出格式不够清晰，就继续调格式。Skill 和代码一样，都是越用越知道该改哪儿。

Skill 是活的，不是写完就供起来的。随着使用次数增加，它会越来越贴合团队的真实需求。

**老王：** 设计 Skill 时，约束程度怎么把握？太严会僵硬，太松会失控。

王哥，这就涉及到一个很关键的设计理念，叫 **自由度分级**。

**高自由度**：纯文本指令，适用于存在多种有效解法的场景。比如“用合适的方式优化这段代码”，AI 就可以有比较大的发挥空间。这类场景你要的是它的思路，不是它的整齐划一。

**中自由度**：伪代码或带参数的脚本，有推荐模式，但允许变化。比如技术方案评审，维度先定好，但可以根据项目特点动态调整权重。说白了，就是“主菜定了，口味允许微调”。

**低自由度**：特定脚本、少量参数，适用于必须严格一致的操作。比如代码格式化，就得老老实实用指定的 `format_code.py`，不能各有各的审美。否则今天是你的风格，明天是他的风格，代码库最后像拼盘。

## 06、Skills 如何提升职场竞争力

**老王：** 最后一个问题。你觉得掌握 Skills 开发能力，对程序员的职场竞争力有什么帮助？

我觉得，至少有三个层面的提升，而且每一层都很实际。

**第一层，效率提升。** 会用 Skills 的程序员，工作效率通常会明显高于不会用的。同样一件事，别人 2 小时，你 20 分钟做完，这差距就是最直接的竞争力。职场里很多时候不比谁更辛苦，而是比谁更快把事做对。

**第二层，经验沉淀。** 普通程序员解决完问题，事情往往就到此为止了，像一桶水倒地上，响一声就没了。会用 Skills 的程序员，会顺手把解决方案沉淀下来，变成可复用的资产。这是从“解决问题”到“沉淀方法论”的跃迁。

**第三层，影响力扩大。** 你写的 Skills 被团队使用，你的经验就会以 Skills 的形式传播开来。这比写文档、做分享更有穿透力，因为 Skills 不是“讲给别人听”，而是“拿来就能用”。很多人分享讲完了，台下热烈鼓掌，回去照样各写各的；但 Skill 一接上，大家就真的开始按这套做了。

![](https://files.mdnice.com/user/3903/820b52fa-bd97-4e5d-aef8-8e4ff5c63438.jpg)

**老王：** 如果让你给初学者一个建议，你会说什么？

别一上来就贪大，先从一个小场景开始。

很多人一开始就想搞一个“万能 Skill”，结果最后写得很复杂，用起来还处处拧巴。像那种功能很多的遥控器，按钮密密麻麻，真到要按的时候反而手足无措。

我的建议是，先观察你工作里那些反复出现的小动作，挑一个最简单的，先把它封装成 Skill。比如“生成 Git 提交信息”“格式化代码注释”。先拿个小场景练手，比空想一个大而全的体系靠谱得多。

先用起来，再慢慢迭代、补功能。Skills 很多时候是长出来的，不是第一天就设计圆满的。真正在团队里活下来的 Skill，往往都带着一点“边用边改”出来的烟火气。

## ending

面试结束了，但关于 Skills 的思考，反倒是从那一刻才真正开始。

Skills 不是什么玄乎的新名词，说到底，它更像是一种新的工作方式。把重复的事情交给 AI，把真正需要判断和创造的部分留给自己。

【**技术的本质不是工具，而是思维方式的升级。Skills 让我们重新思考：什么是值得做的，什么是可以交给 AI 的。**】

如果你还没开始用 Skills，现在就是一个很好的时间点。从一个小场景开始，先写出你的第一个 Skill。

别担心一开始写得不够好，Skills 本来就是迭代出来的。先跑起来，再慢慢优化。
