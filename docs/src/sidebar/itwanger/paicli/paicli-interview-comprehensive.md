## content

### 01、PaiCLI 和 Claude Code 在架构上有哪些相似和不同

这道题拿来开场刚好。面试官不是让你列一份功能清单做表格对比，而是想听你对两个 Agent 产品的架构层面理解。

先说相似点。骨架上高度一致，核心都是一个 ReAct 循环，一个 while 循环把消息历史发给大模型，检查响应里有没有 tool_calls，有就执行工具并把结果塞回历史，没有就退出循环返回最终回复。Claude Code 的主循环大约 1700 行 TypeScript，PaiCLI 用 Java 实现了同样的逻辑，1000 行出头。

Claude Code 在 2026 年 3 月因为 npm 包打包失误泄露过源码，后来有学术团队做了完整分析（VILA-Lab，arxiv 2604.14228），发现只有 1.6% 的代码是 AI 决策逻辑，剩下 98.4% 全是确定性基础设施，包括权限管理、上下文管理、工具路由、恢复逻辑。PaiCLI 的结构比例类似，大量代码花在安全围栏、TUI 渲染、配置管理上。

两者都支持 MCP 协议接入外部工具，都有 stdio 和 HTTP 两种传输。安全层都有人机协同审批机制。记忆系统思路也接近，Claude Code 有 CLAUDE.md 做项目级记忆，PaiCLI 有持久化的长期记忆 JSON 存储。

#### 不同点里你做了哪些设计选择

不同点是面试重点。

| 维度 | Claude Code | PaiCLI |
|---|---|---|
| 语言 | TypeScript | Java 17 |
| 模型 | Claude 系列 | GLM / DeepSeek / Kimi / StepFun |
| 上下文压缩 | 五层 compaction pipeline | 按用户消息边界切分 + LLM 摘要 |
| Multi-Agent | Sub-agent + worktree 隔离 | 三角色协作（Planner/Worker/Reviewer） |
| Plan 模式 | 内置自动规划 | 用户显式 /plan 触发 |
| Skill | hooks + plugins + MCP | 独立注册表，三层加载 |

回答时重点说你的选择理由。比如为什么用显式 /plan 而不是自动路由？PaiCLI 面向教学，显式触发让学习者清楚看到什么时候用 ReAct、什么时候用 Plan-and-Execute。产品化的 Agent 做自动路由没问题，但教学项目需要可见性。

上下文压缩的方案也值得展开。PaiCLI 在用户消息边界切分历史，把旧消息喂给 LLM 生成摘要，保留最近 3 轮对话不动。切分点选在用户消息边界，是为了保证 tool_call 和 tool_result 成对出现，不把一次工具调用拆到两段里。摘要输入上限卡在 60K 字符，防止压缩本身把模型撑爆。

### 02、为什么选 Java 而不是 Python/TypeScript 来做 Agent CLI

技术选型题。面试官不想听你站队"Java 天下第一"，他想看你能不能从实际约束出发逐条分析。

选 Java 有四个具体原因。

**教学目标匹配**。PaiCLI 面向 Java 开发者群体，这帮人天天写 Spring Boot，用 Java 实现 Agent 他们能看懂每一行代码。换 Python 写，读者还得先学 asyncio 那套异步模型，对 Java 程序员来说是额外认知负担。

**类型系统在大项目里真的有用**。PaiCLI 发展到 21 期，28 个包、几十个类之间的依赖关系，靠 IDE 的类型检查和重构工具管理起来很顺畅。Python 的 type hints 是可选的，项目大了容易乱。

**并发原语够用**。JVM 的 ExecutorService、CompletableFuture 开箱即用。做并行工具调用，一个 invokeAll() 就搞定了；异步快照，一个 newSingleThreadExecutor() 就解决了，不需要引入额外框架。

**生态依赖**。JGit 是纯 Java 的 Git 实现，做 Side-History 快照不依赖本机 git 命令。JavaParser 做 AST 分析支撑 RAG 分块，Java 生态在这块有成熟方案。

【此处插入依赖截图：截图目标：PaiCLI 的 pom.xml 展示 JGit、JavaParser 等关键依赖；关键词：pom.xml、JGit、JavaParser；建议位置：命令行】

#### Python 和 TypeScript 的优势你怎么看

公正分析一下对手。Python 的 AI/ML 生态最成熟，LangChain、LlamaIndex 这些框架开箱就用，写原型最快。TypeScript 是 Claude Code 本身的语言，MCP SDK 的 TypeScript 版本最完整，和前端衔接方便。

没有绝对对错。选型取决于团队背景、项目目标和生态需求。面试官想听的是你能不能列出约束条件、逐条分析、给出有理有据的结论，而不是争论哪个语言更好。

### 03、如果从零设计一个 Agent 产品，你怎么规划迭代路径

这道题考的是产品规划能力。不是写 PPT，而是真正做过从 0 到 1 的人才答得出来的东西。

参考 PaiCLI 的 21 期演进，浓缩成四个阶段。

**Phase 1：最小可用（1-2 周）**。目标就一个，跑通。ReAct 循环 + 基础工具（读写文件、执行命令）+ 单模型接入 + 纯文本输出。PaiCLI 第一期就是这个状态，400 行代码，能对话、能调工具，够了。

**Phase 2：核心能力（2-4 周）**。这一步加的都是没有就不能上生产的东西：

- Plan-and-Execute 模式，让 Agent 先规划再执行
- Memory 系统，短期对话 + 长期事实
- 人机协同审批 + 安全围栏
- 多模型适配，运行时切换

**Phase 3：生态集成（2-4 周）**。接入外部世界，MCP 协议（stdio + HTTP）、联网搜索与抓取、浏览器操控。这个阶段 Agent 的能力边界一下子扩大了。

**Phase 4：产品化（2-4 周）**。让用户真正用得起来：TUI 流式渲染、LSP 诊断注入、Git 快照与回滚、后台任务 + HTTP API。

【此处插入迭代路径时间线：截图目标：PaiCLI 从 Phase 1 到 Phase 4 的 21 期演进路径；关键词：ROADMAP、21 期、迭代；建议位置：白板】

#### 迭代过程中最重要的原则是什么

**先跑通再优化**。Phase 1 的 400 行代码能跑，比一上来就设计完美架构重要得多。我见过不少人还没写第一行代码就在纠结分层和框架选型，结果一行有效代码都没产出。

**安全从第二步就加**。改造成本随时间指数上升。PaiCLI 在第六期就上了人机协同审批、路径围栏和命令黑名单，后面所有新工具自动受这层保护。如果安全拖到第 20 期才补，前面 19 期的工具全要回头改。

**工具决定能力边界**。Agent 能做什么 = 它有什么工具。优先做用户最需要的工具，别为了"架构完整"去做没人用的东西。

### 04、Agent 的可观测性怎么设计

生产级 Agent 必须回答三个问题：它在干什么？花了多少钱？出问题了怎么排查？

PaiCLI 的可观测性分五层。

**第一层：运行时状态栏**。底部状态栏实时显示当前模型、Token 用量、审批状态、MCP Server 连接数。/context 命令输出上下文模式、Prompt Cache 命中率、RAG topK 配置。用户随时能看到 Agent 的"仪表盘"。

**第二层：成本追踪**。每轮输出 Token 用量和估算成本（人民币）。区分普通 input、cached input 和 output 三种类型分别计价。cached input 通常只有普通 input 价格的十分之一甚至二十分之一，不区分的话成本估算偏差很大。DeepSeek V4 的价格是 input ¥2/百万、cached ¥0.5/百万、output ¥8/百万。

**第三层：审计日志**。采用 JSONL 格式，一行一条 JSON，按天分文件存储在 `~/.paicli/audit/` 目录。每条记录包含时间戳、工具名、参数（脱敏后）、执行结果（allow/deny/error）、审批来源、执行耗时。敏感信息比如 Bearer Token 和 API Key 会用正则匹配后替换成 `***`，超过 1000 字符的字段自动截断。写入失败只在 stderr 提示，不抛异常。审计故障不能影响主流程。

**第四层：调试日志**。写在 `~/.paicli/logs/paicli.log`，LLM 返回的 reasoning_content（深度思考过程）写日志但不进对话历史，避免白白占 Token 预算。

**第五层：对话历史**。完整的对话记录存在 `~/.paicli/history/` 下，JSONL 格式，可重放、可分析。

#### 你们的审计日志设计意图是什么

把 Agent 的"实际副作用"变成可回放的事实流。行为评估、差错复盘、监控告警都从这一条数据流里出。面试时把这五层按层次讲清楚就够了，面试官真正看重的是你有没有"可观测性分层"的意识，而不是具体实现细节。

### 05、怎么处理 LLM 的幻觉问题

Agent 场景下幻觉不只是"回答不准确"。LLM 编造一个不存在的文件路径，Agent 真去读了，读失败了，LLM 又编造另一个路径，Agent 又去读。这就是幻觉驱动的死循环。

典型表现有三种：编造不存在的文件路径、编造不存在的 API 或函数名、对工具返回结果做错误解读。

PaiCLI 的应对策略，核心就一句话：**不信任 LLM 的陈述，信任工具返回的事实**。

整个 ReAct 循环的设计哲学就是围绕这句话展开的。LLM 说"这个文件应该有某个函数"，但 read_file 返回的内容里没有，Agent 会把真实内容喂给 LLM，让它基于事实重新推理，而不是继续相信自己的"记忆"。

具体来说有五道防线：

- **工具结果为准**：推理基于工具返回的真实数据，不基于 LLM 的先验知识
- **RAG 注入真实代码**：向量检索 + 关键词检索的混合模式，把真实代码段摆在 LLM 面前，减少凭空想象的空间
- **LSP 诊断兜底**：LLM 编造了不存在的类名写进代码，LSP 诊断报编译错误，错误信息注入下一轮对话让 LLM 自动修复。这层很重要，把"运行时才发现的错"提前到了"编辑时就能发现"
- **Reviewer 角色审查**：Multi-Agent 模式下 Reviewer 对 Worker 的输出做质量审查，相当于一层代码审查
- **用户确认**：人机协同审批让用户在关键操作前检查 LLM 的输出是否合理。人是最后一道防线

面试时把那句话说出来就够了，**不信任陈述，信任事实**。这个原则能覆盖 80% 的幻觉场景。

### 06、Agent 的错误恢复策略有哪些

Agent 不是跑一次就完事的脚本，它是一个持续运行的循环。任何一个环节出问题都不应该让整个 Agent 挂掉。

按严重程度分四级。

**轻度：工具执行失败**。最常见的情况。把错误信息直接返回给 LLM，让它自己决定重试还是换方案。比如读文件时路径不存在，LLM 拿到"文件不存在"的错误后，通常会调目录列表工具找正确路径。单个工具失败不影响其他并行工具的执行。

**中度：MCP Server 挂了**。该 Server 注册的工具暂时不可用，Agent 可以 fallback 到内置工具替代。比如 Chrome DevTools MCP 挂了，降级到内置的网页抓取工具。/mcp restart 可以手动重启，重启后自动重新注册工具。

**重度：LLM API 超时或限流**。流式响应中断时，已收到的部分输出会保留，不会因为最后一个 chunk 超时就丢掉前面所有内容。限流有退避策略。API 完全不可用时，支持运行时切到另一个模型，四个模型提供商之间可以热切换。

**致命：Agent 改坏了文件**。这是最可怕的。PaiCLI 在每轮 Agent 执行前后都用 JGit 做 Git Side-History 快照，/restore 命令可以回滚到任意快照点。甚至还有一个 revert_turn 工具，允许 Agent 自己触发回滚。当它发现自己改错了，不用等用户手动干预。

【此处插入快照回滚截图：截图目标：/restore 命令列出快照列表并选择回滚的终端交互；关键词：restore、snapshot、回滚；建议位置：命令行】

#### 停滞检测是怎么做的

除了错误恢复，还有一种隐蔽的故障，Agent 陷入死循环。PaiCLI 在 Token 预算管理里内置了停滞检测：如果连续 3 次工具调用的签名完全一致（工具名 + 参数 hash），判定为停滞，强制退出循环。另外还有一个 50 轮的硬限，防止极端情况。Claude Code 也有类似的机制，连续 3 次回复少于 500 Token 就判定为收益递减，主动停下来。

### 07、手写 Agent 和用框架有什么区别

| 维度 | 手写 | 框架（LangChain / Spring AI） |
|---|---|---|
| 学习成本 | 理解原理，但开发慢 | 上手快，但黑盒多 |
| 灵活性 | 完全可控 | 受框架约束 |
| 调试 | 断点直接查 | 抽象层多，调试困难 |
| 依赖 | 最小化 | 大量传递依赖 |
| 升级 | 自己维护 | 框架 breaking change |
| 适合场景 | 教学、深度定制 | 快速原型、标准场景 |

PaiCLI 选手写的原因很直接，教学项目需要学生看懂每一行代码。用 Spring AI 写个 Agent，三行注解就跑了，但学生完全不知道底下发生了什么。ReAct 循环在哪？tool_calls 怎么解析的？Token 预算怎么管理的？全藏在框架里。

PaiCLI 的主循环从第一行 while(true) 到最后一行 return，中间经历了什么一目了然。这对理解 Agent 原理的价值是框架给不了的。

先说手写的好处（理解原理、完全可控），再说框架的好处（效率高、生态丰富），最后说你的选择逻辑，快速验证用框架，需要深度定制或教学用手写。别站队，面试官不想听极端观点。

### 08、Agent 的测试策略是什么

最大的难点是 LLM 输出非确定性。同样的输入跑两次，模型可能返回不同的工具调用顺序、不同的推理路径。你不能像测一个纯函数那样断言输出等于某个固定值。

PaiCLI 的测试分四层。

**第一层：单元测试**。把能确定性测试的部分独立出来。路径围栏的规则匹配、命令黑名单检查、JSON-RPC 解析、SSE 增量合并、Token 预算计算，这些全是纯函数，输入确定输出确定，标准 JUnit 搞定。PaiCLI 有 457+ 测试用例覆盖这批组件。

**第二层：集成测试**。MCP Server 启动和工具注册、审批流程、配置加载优先级（四级加载顺序：默认 → 全局 → 项目 → 运行时）。涉及多组件协作但不依赖 LLM，结果确定。

**第三层：端到端测试**。完整的 ReAct 循环需要 API Key，浏览器 MCP 需要本机 Chrome。这类不在 CI 里自动跑，手工验证。

**第四层：Prompt 回归测试**。固定一组测试用例 + 预期行为，改了 Prompt 模板或系统提示词后跑一遍，人工比对输出有没有偏差。

核心原则就一条：**把能确定性测试的部分独立出来测**。Agent 循环是非确定性的，但循环里的每个组件，比如工具执行、路径校验、审计记录、Token 计算，都是确定性的。

### 09、如何向面试官介绍你的 Agent 项目

这是面试最高频的问题，但大多数人讲不好。要么讲太细面试官听不下去，要么讲太泛没有技术含量。

1 分钟版本要做到有骨架、有亮点、有数据。参考话术：

### 10、你在做 PaiCLI 的过程中踩过哪些坑

面试官爱问踩坑经历，具体的坑比成功经验更能证明你是真做过的。

**坑 1：工具描述太模糊导致 LLM 乱选工具**。早期命令执行工具的描述写得太泛，LLM 经常用 cat 命令代替 read_file 读文件。排查了半天才意识到问题出在工具描述而不是模型能力上。解决方法很土但有效，在描述里明确写"不要用此工具读文件"。这个教训的本质是：Function Calling 的质量取决于工具描述的质量。

**坑 2：SSE 增量 tool_calls 解析**。流式响应里的 tool_calls 是增量的，每个 chunk 只有一部分 JSON，不是完整的。一开始我以为每个 chunk 都是完整的 tool_call，解析直接报错。查了好久才搞清楚，需要按 index 字段累积拼接函数名和参数字符串，全部 chunk 到齐后再做一次完整的 JSON 反序列化。

**坑 3：符号链接绕过路径校验**。路径围栏早期用 getCanonicalPath() 校验，但如果项目内有一个指向 /etc/ 的符号链接，canonical path 解析后就绕过了围栏。后来改用 Java NIO 的 toRealPath() 并且对展开后的真实路径做二次校验才堵住。macOS 上还有个隐蔽的坑，/var 是 /private/var 的符号链接，项目根目录本身就需要预展开。

**坑 4：MCP Server 启动超时**。Chrome DevTools MCP 首次启动要用 npx 下载依赖，默认 30 秒超时不够。用户一启动就报超时，以为是配置错了。解决方案是超时提升到 60 秒，加进度提示让用户知道在等什么。

### 11、如果要把 PaiCLI 部署到生产环境，还需要做什么

教学项目和生产产品之间差距是巨大的。写一个能跑的 Agent 和运营一个稳定的 Agent 服务，完全是两码事。

需要补的东西清单：

- **沙箱隔离**：当前命令执行工具直接操作宿主文件系统，生产环境必须用 Docker 或 microVM 隔离。路径围栏是软防护，容器是硬隔离
- **多租户**：当前是单用户的。生产环境每个用户需要独立的对话历史、长期记忆和配置，互不干扰
- **限流和配额**：按用户限制 LLM 调用次数和 Token 消耗。当前的预算管理只管单次运行，不管累计消费
- **监控告警**：Prometheus + Grafana 监控 Agent 延迟、错误率、Token 消耗。审计日志的 JSONL 可以作为告警数据源，但需要采集管道
- **日志聚合**：当前日志在本地文件。多节点部署需要 ELK 或 Loki 做集中式日志
- **认证鉴权**：Runtime API 当前用简单 API Key，生产环境需要 OAuth / JWT + RBAC
- **模型网关**：统一的 LLM 调用网关做负载均衡、fallback 和成本控制，当前的进程内切换不够

面试时不用把每一项都展开讲，挑两三个你理解最深的说清楚就行。面试官真正想考的是你有没有"教学原型 vs 生产系统"的认知差距意识。

### 12、Agent 领域你觉得下一个大方向是什么

开放讨论题，面试官想看你对行业趋势有没有自己的判断。别背新闻稿，说你真正关注的方向。

**Computer Use / GUI Agent**。Agent 不只操作 API 和命令行，而是能像人一样操作 GUI，点击按钮、填表单、拖拽文件。PaiCLI 集成 Chrome DevTools 协议做浏览器操控已经在做这件事了，只不过是通过协议接口而不是像素级操作。Anthropic 的 Computer Use、Google 的 Project Mariner 代表了更激进的方向，直接看屏幕截图来决策。

**多 Agent 协作标准化**。Google 的 A2A（Agent-to-Agent）协议在 2025 年 4 月发布，到 2026 年 3 月已经更新到 v1.2，超过 150 家组织支持，包括 Google、Microsoft、AWS、Salesforce。它已经归入 Linux Foundation 管理，规范变更走公开 RFC 流程。如果这个标准化成了，不同厂商的 Agent 就能互相发现、互相委派任务。PaiCLI 的三角色模型是自定义的，未来可以考虑对接 A2A。

**端到端评估**。怎么系统化评估 Agent 完成复杂任务的能力？SWE-bench Verified 是目前最主流的基准，截至 2026 年 6 月，最高分是 Claude Mythos Preview 的 93.9%，但更难的 SWE-bench Pro 上最高也就 45.9%。83 个被评估的模型平均分 63.4%，头尾差距 30 个百分点。评估体系还远远不够成熟。

**低成本推理**。Agent 的 Token 消耗远超普通对话。一个 Plan-and-Execute 任务可能消耗几十万 Token。推理成本降低（更高效的 KV Cache、推测解码、大小模型混合路由）直接影响 Agent 的商业可行性。PaiCLI 的 Prompt Cache 在 DeepSeek V4 上能把缓存命中部分的成本降到原来的四分之一，但还不够。

### 13、你觉得 AI Agent 会取代程序员吗

面试官偶尔会问这种开放题。没有对错，但你的回答能暴露思考深度。

不会取代，但会重新定义"程序员"的日常。

Agent 擅长的事情已经很多了，模板化工作（CRUD、配置修改、测试用例生成）、搜索和检索（找 API、查文档、读代码）、重复性重构（批量重命名、格式统一、迁移）。这些体力活确实会被接管，速度会比大多数人预期的快。SWE-bench Verified 最高分已经到 93.9%，500 道真实 GitHub issue 里有 469 道能自动解决。

但 Agent 不擅长的事情同样明确。

**理解模糊需求**。客户说"做个像抖音一样的"，Agent 不知道该怎么开始。人能通过追问、推测、类比把模糊需求转化成具体方案，这个能力还差得远。

**架构决策**。微服务还是单体、SQL 还是 NoSQL、自建还是买服务，这些决策需要理解业务上下文、组织约束和长期演进方向。PaiCLI 选 Java 而不是 Python 的决策过程，Agent 做不了。

**在约束中创造**。做 PaiCLI 的 TUI 时，我反复在 Claude Code 的交互上找灵感、在终端限制中做取舍。这种"在限制里寻找最优解"的过程，目前的 Agent 能力远远不够。

未来的程序员更像是 Agent 的指挥者，定义目标、审核输出、处理 Agent 搞不定的复杂决策。日常编码的体力活交给 Agent，人专注在更高层的思考上。

用 PaiCLI 的话来说：你是用户，Agent 是执行者。**人机协同审批不会消失，因为最终的决策权必须在人手里。**
