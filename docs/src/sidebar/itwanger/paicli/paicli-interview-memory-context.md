上一篇我们聊了 Agent 核心架构（ReAct、Plan-and-Execute、Multi-Agent），这篇进入第二个高频面试方向：**记忆与上下文工程**。

这块对应 PaiCLI 的第 3 期（Memory 系统）、第 4 期（RAG 检索）和第 12 期（长上下文工程），是 Agent 能不能"记住事情"、"理解代码库"、"处理长对话"的关键。

## 01、Agent 的记忆系统分哪几层？各自解决什么问题？

典型的 Agent 记忆系统分三层：

- **短期记忆（Working Memory）**：就是当前对话的消息历史。每轮用户输入、LLM 回复、工具调用和结果都存在这里。受限于上下文窗口大小，对话太长就装不下了。
- **长期记忆（Long-term Memory）**：跨会话持久化的关键事实。比如"用户偏好中文回复"、"项目用的是 Spring Boot 3"。存在文件或数据库里，新会话启动时检索相关记忆注入 system prompt。
- **外部记忆（External Memory）**：通过 RAG 检索访问的外部知识（代码库、文档库）。不直接放在对话历史里，而是按需检索相关片段注入。

PaiCLI 的实现：
- 短期记忆：`ConversationMemory` 维护当前会话的消息列表
- 长期记忆：`LongTermMemory` 持久化到 `~/.paicli/memory/long_term_memory.json`
- 外部记忆：`CodeRetriever` 通过向量检索访问代码索引

## 02、短期记忆会溢出吗？怎么处理？

会。LLM 有上下文窗口限制（200k–1M tokens），短期记忆不做管理的话，对话历史会不断膨胀直到超出窗口。

PaiCLI 的解决方案是**摘要压缩**：

1. `TokenBudget` 实时跟踪当前对话占用的 token 数
2. 当占用接近预算阈值时（默认 80% 窗口），触发 `ContextCompressor`
3. `ContextCompressor` 把早期对话批量发给 LLM，让 LLM 生成一段精炼摘要
4. 用摘要替代原始的多轮对话，释放 token 空间

摘要压缩的策略采用 Map-Reduce：先把长对话分段摘要（Map），再把多段摘要合并成一个总摘要（Reduce）。

关键权衡：**压缩越激进，丢信息越多**。PaiCLI 只压缩早期对话，最近几轮保持原文不动——因为最近几轮通常是当前任务的关键上下文。

## 03、长期记忆什么时候存、什么时候取？

**存（写入）的时机**：

- 用户显式指令：`/save 这个项目用的是 Java 17` 或用户说"记一下"
- Agent 主动调用 `save_memory` 工具：当用户明确要求保存偏好或稳定事实时

**取（检索）的时机**：

- 每轮对话开始前，`MemoryRetriever` 对用户输入做语义匹配
- 匹配到相关记忆后，作为上下文注入到 system prompt 或 user message 前面
- 检索用关键词匹配 + 简单的相似度评分

PaiCLI 的设计原则是**只存稳定事实，不存临时信息**。"用户偏好中文"可以存，"当前正在改 Main.java"不应该存——后者是短期记忆的事。

## 04、什么是 RAG？为什么 Agent 需要 RAG？

RAG（Retrieval-Augmented Generation）= 检索增强生成。核心思路是：LLM 回答之前，先从外部知识库检索相关内容，把检索结果塞进 prompt 一起发给 LLM。

Agent 需要 RAG 的原因：

1. **LLM 不认识你的代码库**。LLM 的训练数据里没有你的项目代码，不检索就是盲人摸象。
2. **全量代码塞不进上下文**。一个中型 Java 项目 10 万行代码，远超任何模型的上下文窗口。
3. **检索比暴力塞更高效**。只把相关代码段塞进 prompt，模型注意力更集中、成本更低、回答更准。

PaiCLI 的 RAG 流程：
```
用户问题 → Embedding → 向量检索 → Top-K 代码段 → 注入 prompt → LLM 回答
```

## 05、代码 Embedding 是怎么做的？有什么讲究？

Embedding 就是把代码文本转成固定长度的数字向量，语义相近的代码段在向量空间里距离近。

PaiCLI 的做法：

1. **代码分块（Chunking）**：`CodeChunker` 把代码按文件、类、方法三个粒度拆成块。不是简单按行数切，而是用 JavaParser 做 AST 解析，确保每个块是一个完整的语义单元。
2. **向量化**：把每个代码块发给 Embedding 模型（默认用本地 Ollama 的 `nomic-embed-text`），得到向量。
3. **存储**：向量存到 SQLite 数据库里（`~/.paicli/rag/codebase.db`）。

分块的讲究：

- **粒度太粗**（整个文件一块）：一个文件包含多个不相关的类/方法，检索会把无关内容也带出来。
- **粒度太细**（单行一块）：丢失上下文，一行代码脱离前后文毫无意义。
- **最佳实践**：方法级粒度，每个块是一个完整方法 + 所属类名 + import 信息。

## 06、向量检索用的什么算法？为什么选余弦相似度？

PaiCLI 用的是**余弦相似度（Cosine Similarity）**：

```
cosine_sim(A, B) = (A · B) / (|A| × |B|)
```

值域是 [-1, 1]，1 表示方向完全相同（最相似），0 表示正交（无关），-1 表示方向完全相反。

为什么选余弦而不是欧氏距离？

- **不受向量长度影响**：余弦只看方向不看大小。两段代码的 Embedding 长度可能不同（取决于文本长度），余弦不受干扰。
- **高维空间效果好**：Embedding 通常是 768 或 1536 维的高维向量，欧氏距离在高维空间会出现"维度灾难"（所有点距离趋于相同），余弦更稳定。
- **计算简单**：点积 + 两次向量模长，SQLite 里都能算。

PaiCLI 没有引入专门的向量数据库（Milvus、Pinecone），而是直接在 SQLite 里存向量、用 Java 计算余弦相似度。对于中小型代码库（< 10 万行），这个方案够用且零依赖。

## 07、AST 解析在 RAG 里起什么作用？

AST（Abstract Syntax Tree，抽象语法树）是代码的结构化表示。PaiCLI 用 JavaParser 解析 Java 代码得到 AST，主要干两件事：

**1. 智能分块**：根据 AST 节点类型（ClassDeclaration、MethodDeclaration）切分代码块，而不是简单按行数切。保证每个块是一个语义完整的单元。

**2. 代码关系图谱**：`CodeAnalyzer` 从 AST 中提取类之间的关系——extends、implements、imports、method calls、contains。这些关系存到 `CodeRelation` 表里，用户查 `/graph ClassName` 就能看到这个类的继承、实现和调用关系。

关系图谱的价值：Agent 搜代码时不仅能找到直接相关的代码段，还能顺着关系链找到上下游依赖。比如你问"谁调用了 Agent.run()"，图谱能直接给出答案，不需要全文搜索。

## 08、长上下文模型（200k-1M）出来后，RAG 还有必要吗？

有必要，但角色变了。

**短上下文时代（4k-32k）**：RAG 是必须的，代码库塞不进去，必须检索。
**长上下文时代（200k-1M）**：小型项目可以整个塞进去，但 RAG 仍有价值。

RAG 在长上下文时代的价值：

1. **成本控制**：100k token 的 input 比 10k token 贵 10 倍。RAG 精准检索能大幅降低输入成本。
2. **注意力精度**：模型对长文本的中间部分注意力会下降（"Lost in the Middle" 问题）。RAG 把相关内容前置，模型更容易抓住重点。
3. **加速响应**：输入越长，首 token 延迟越高。RAG 精简输入能显著加速。
4. **超大代码库**：50 万行代码的 monorepo，1M 窗口也装不下。

PaiCLI 第 12 期的策略是**按模式自适应**：短模式 topK=5，balanced 模式 topK=10，长模式 topK=20。长窗口不是不用 RAG，而是可以多检索一些给模型更完整的上下文。

## 09、Prompt Caching 是什么？对 Agent 有什么影响？

Prompt Caching 是模型提供商的优化——如果连续请求的 prompt 前缀相同，服务端可以复用上一次的 KV Cache，跳过重复计算。

**对 Agent 的影响**：

- **成本降低**：cached input tokens 的价格通常只有普通 input tokens 的 1/5 到 1/10。
- **速度提升**：跳过前缀计算，首 token 延迟更低。
- **prompt 布局要求**：要最大化 cache 命中率，**稳定内容放前面、变化内容放后面**。

PaiCLI 的 prompt 布局策略（第 19 期 `PromptAssembler`）：

```
base.md（稳定，几乎不变） → personality（稳定） → mode（按模式切换） → approval（HITL 配置）
→ project_context（项目相关） → skills（按需加载） → context_mgmt（动态） → handoff（最后）
```

越往前越稳定，越往后越动态。这样前面的部分能持续命中 KV Cache，只有后面的动态部分需要重新计算。

## 10、上下文压缩有哪些策略？各自的优缺点？

常见的上下文压缩策略：

**1. 截断法**：直接丢弃最早的对话。最简单但最粗暴，可能丢失关键背景。

**2. 摘要法**：用 LLM 对早期对话生成摘要，用摘要替代原文。PaiCLI 用的就是这种（Map-Reduce 摘要）。优点是保留语义，缺点是摘要本身消耗一次 LLM 调用。

**3. 选择性保留**：只保留 system prompt + 最近 N 轮对话 + 所有工具调用结果。中间的"闲聊"丢掉。需要判断哪些是"闲聊"，实现复杂。

**4. 分层存储**：近期对话保持原文，中期对话做摘要，远期对话只保留关键事实到长期记忆。最精细但实现成本最高。

PaiCLI 的选择是**摘要法 + 长上下文模式跳过**。理由：

- 摘要法在短窗口（< 100k）场景下效果好
- 长窗口（≥ 100k）直接不压缩，省掉摘要的 LLM 调用成本
- 架构简洁，两种策略覆盖所有场景

## 11、代码索引的更新策略是什么？代码改了怎么同步？

PaiCLI 的代码索引目前是**手动触发**的——用户执行 `/index` 命令时全量重建索引。

这是一个有意的 MVP 选择，更成熟的方案有：

**1. 增量索引**：监听文件系统变更（Java WatchService），只对修改的文件重新 Embedding。优点是实时，缺点是 WatchService 在 macOS 上不够可靠，大规模文件变更时可能漏事件。

**2. Git diff 索引**：每次 Agent 启动时，对比上次索引的 commit hash 和当前 HEAD，只重索引 diff 涉及的文件。比文件系统监听更可靠。

**3. 延迟索引**：Agent 搜索时发现结果过时（文件 mtime > 索引 mtime），触发单文件重索引。按需更新，不浪费资源。

对于面试回答，建议提到"PaiCLI 当前是手动全量，生产环境应该做增量 + Git diff"。

## 12、记忆检索的质量怎么评估？

记忆检索的质量评估可以从两个维度看：

**精确率（Precision）**：检索出来的记忆里，有多少是真正相关的。比如检索 5 条，3 条相关，精确率 60%。

**召回率（Recall）**：所有相关记忆里，有多少被检索出来了。比如实际有 10 条相关，检索出了 3 条，召回率 30%。

Agent 场景更看重**精确率**——塞无关记忆进 prompt 会干扰模型，不如少塞但塞准。

PaiCLI 的检索策略偏保守：

- 长期记忆检索用关键词匹配 + 相似度评分，只返回高于阈值的结果
- RAG 代码检索限制 topK，不贪多
- 检索结果放在 prompt 特定位置，和正文对话隔开

## 13、如果面试官问"你的记忆系统和 ChatGPT 的 Memory 有什么区别"，怎么回答？

核心区别在**应用场景和存储粒度**：

| 维度 | ChatGPT Memory | PaiCLI Memory |
|---|---|---|
| 场景 | 通用对话助手 | 代码工程 Agent |
| 存储内容 | 用户偏好、个人信息 | 项目事实、技术偏好、代码约定 |
| 检索方式 | 全量注入（记忆少时）| 语义匹配 + 按需注入 |
| 与工具协同 | 无 | 记忆指导工具选择（如"这个项目用 Maven 不用 Gradle"）|
| 外部记忆 | 无 | RAG 代码检索 |

PaiCLI 的记忆系统更接近 Claude Code 的 CLAUDE.md 机制——项目级的稳定事实持久化，而不是泛泛的"用户画像"。

回答时强调"Agent 的记忆不仅用于对话，还直接影响工具调用决策"，这是和纯聊天场景最大的不同。
